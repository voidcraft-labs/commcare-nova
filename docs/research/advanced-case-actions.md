# Advanced Case Actions (Save to Case): what it is, why it exists, when to reach for it

*Research memo, July 2026. Sources: the Dimagi codebases (`commcare-hq`, `commcare-core`, `formplayer`, `Vellum`, all at `~/code/`), the Confluence dump (`.data/confluence-cache/pages`, snapshot ≈ Feb 2026), and two production Colorado care-coordination apps — **Client Care Search** (v13396) and **Central Registry** (v7331) from the `co-carecoordination-dev` project space. Code citations are `repo/path::symbol`; app citations are form name first, then index, e.g. "Create Profile and Refer (m2-f0)", with the backing file in the unzipped `.ccz`.*

**TL;DR.** ACA lets a form mutate *any* case — create, update, close, or re-link — identified by an expression rather than by the session's case selection. Mechanically it is nothing more than authoring raw case-transaction XML blocks inside the form; every form's case work compiles to the same blocks. Its reason to exist is CommCare's ownership model: who *owns* a case determines who *syncs* it, sync size determines performance, and therefore serious apps keep users owning almost nothing and reach everything else through live search and claim. In that world the case a form must write to is routinely *not* the case the session selected — it lives in another location's bucket, or belongs to a colleague, or doesn't exist yet and must be born owned by someone else. ACA is the write-side instrument for that architecture, and the two Colorado apps run almost their entire data model through it.

---

## 0. A note on names

The feature goes by two names:

- **"Save to Case"** — the feature's name for its first decade, and still its name in code. The Vellum plugin (`Vellum/src/saveToCase.js`, mug class `SaveToCase`, first commit April 2015) and all HQ code (`save_to_case`, privilege `VELLUM_SAVE_TO_CASE` in `commcare-hq/corehq/privileges.py`) use it, as do most Confluence pages and the two production apps' own node names.
- **"Advanced Case Actions"** — the feature's *current* product name, and this memo's term (abbreviated **ACA**). A major Vellum UI rework landed **April–June 2026** (git history of `saveToCase.js`); the question type is now titled "Advanced Case Actions" and its help link points to a new Confluence page, "Enabling and Using Advanced Case Actions". The Confluence dump predates that page.

## 1. What ACA is, mechanically

### 1.1 The wire primitive: the case transaction block

Everything CommCare does to cases — from the plainest registration form to the wildest ACA workflow — is expressed on the wire as `<case>` blocks in the namespace `http://commcarehq.org/case/transaction/v2`, embedded in the submitted form instance:

```xml
<case case_id="…" date_modified="…" user_id="…"
      xmlns="http://commcarehq.org/case/transaction/v2">
  <create>
    <case_type>referral</case_type>
    <case_name>Referral</case_name>
    <owner_id>…</owner_id>          <!-- optional -->
  </create>
  <update>
    <any_property>…</any_property>  <!-- arbitrary keys -->
  </update>
  <index>
    <parent case_type="client" relationship="extension">…case-id…</parent>
  </index>
  <close/>
</case>
```

Both the server and the client scan the *entire* submitted instance for these blocks — position in the form tree is irrelevant (`commcare-hq/corehq/ex-submodules/casexml/apps/case/xform.py::get_case_updates` via recursive `extract_case_blocks`; client side `commcare-core/src/main/java/org/commcare/core/parse/CommCareTransactionParserFactory.getParser`, dispatching on element name `case`). This one fact is what makes ACA possible at all: a block inside a repeat group emits once per repeat instance, and a block anywhere in the tree is processed like any other.

Ordinary case management is just HQ *generating* these blocks at app-build time from module/form configuration (at fixed paths like `/data/case`, `/data/subcase_0/case`, `/data/commcare_usercase/case`). ACA is the app author writing the blocks *by hand in the form builder*, which removes every constraint the generator imposed — most importantly, the author controls `@case_id`.

### 1.2 Semantics of the five actions

Verified against the client parser, `commcare-core/src/main/java/org/commcare/xml/CaseXmlParser.java` (actions are processed in document order within a block; multiple blocks for the same case merge sequentially):

| Action | Semantics | Sharp edges |
|---|---|---|
| `create` | Creates the case with the block's `@case_id`. Children are restricted to exactly `case_type`, `case_name`, `owner_id`; the first two are mandatory (`createCase`, "One of [case_type, case_name] is missing"). `owner_id` falls back to the block's `@user_id`; if both are empty the block is rejected. `date_opened` ← `@date_modified`. | Re-creating an existing id silently overwrites name/type rather than erroring (`acceptCreateOverwrites`). All other properties go in a sibling `<update>`. Server-side, a create that never sets an owner gets `owner_id = user_id` (`commcare-hq/corehq/form_processor/backends/sql/update_strategy.py::SqlCaseUpdateStrategy._apply_case_update`). |
| `update` | Sets arbitrary properties. Reserved keys with special behavior: `case_type`, `case_name`, `date_opened`, `external_id`, `owner_id` (changing it triggers a sync-relevance recalculation), plus newer `category`/`state`. | Yes: ACA can **rename a case, re-type a case, and transfer ownership** as ordinary property writes. The public doc advertises re-typing explicitly ("Change a case's type dynamically" — "Enabling Custom Case Management Workflows with Save to Case", commcarepublic). |
| `close` | Marks the case closed. | Closing a host case abandons all its extension cases at the next sync (see §2.2). |
| `index` | `<index><NAME case_type="T" relationship="child\|extension">target-id</NAME></index>`. The *element name* is the index identifier — "parent" and "host" are pure convention, not enforced ("Extension Cases", GS space); the apps use `parent` for extensions throughout, which per the builders was simply the comfortable word, not a compatibility calculation. Setting an existing identifier **replaces** that relationship; an **empty target removes it**; a case indexing itself is an error; relationship defaults to `child`. | Index is a full CRUD surface on the case graph: link, re-link, and unlink. The empty-target-removes rule is load-bearing in production (§3, "break index" blocks). |
| `attachment` | Attach/remove binaries by name (`src`/`from` attributes; both empty = remove). | Exists on the wire and in older docs; the reworked Vellum UI does not surface it. |

Two attributes matter operationally: `@date_modified` (mandatory; conflict resolution) and `@user_id` (who did it; also the owner fallback above).

### 1.3 What the Vellum question actually is

An ACA question ("Advanced Case Actions", under *Advanced* question types) is a data-only node the author names freely; Vellum plants the literal `<case>` template underneath it and expresses all configuration as `<bind>`s over the template's leaves plus, for creates outside repeats, one `<setvalue event="xforms-ready">` for `@case_id` (`Vellum/src/saveToCase.js::getBindList`/`getSetValues`; canonical output in `Vellum/tests/static/saveToCase/*.xml`). From "Create Profile and Refer (m2-f0)" in Client Care Search (`modules-2/forms-0.xml`), a real create looks like:

- the node `/data/client_profile_group/client_profile_save_to_case/create_client_profile/case` holding empty `create`/`update` children;
- one bind per slot: `…/case/create/case_type` `calculate="'client'"`, `…/case/create/owner_id` `calculate=/data/…/client_registry_id`, twenty-eight `…/case/update/<prop>` binds, `…/case/@date_modified` `calculate=/data/meta/timeEnd`, `…/case/@user_id` `calculate=instance('commcaresession')/session/context/userid`;
- `<setvalue event="xforms-ready" ref="…/case/@case_id" value="/data/…/client_case_id"/>` where `client_case_id` was itself seeded with `uuid()`.

The per-question configuration surface (post-rework):

- **Case Actions** chips: Create / Update / Close / **Link–Unlink** (the UI's name for `index`). Create and Update are mutually exclusive *in the UI only* — a create's extra properties still emit as a wire `<update>`, and the parser folds legacy create+update forms together (`mergeUpdatePropertiesIntoCreateAfterParse`).
- **Case ID** — a required, free-form XPath. `uuid()` is allowed only when creating; otherwise the expression must resolve to an existing case's id. This one field is the entire targeting model.
- **Case Type** — a dropdown fed by HQ's data dictionary (`existingCaseTypes`), free-entry only when creating, with an XPath escape hatch; `commcare-user` may only be updated; `user-owner-mapping-case` is reserved.
- **Create**: open-case condition (relevant on the whole block), case name, owner id (+ its own condition), property cards. **Update**: property cards, each `{name, calculate, relevant}`. **Close**: close condition. **Link/Unlink**: cards of `{identifier, relationship, host-case-id, host-case-type}` — leaving the host id empty is how you *unlink*.
- **Advanced** (collapsed): `date_modified` and `user_id` expressions, pre-filled with the defaults above.

One idiom the configuration surface doesn't show: **whole-block conditionality**. Create has its open-case condition, close has its close condition, and each property row takes a `relevant` — but there is no "update condition" gating an entire update-only block (an asymmetry that survived the rework). The production workaround is structural: wrap the ACA question in a group whose display condition is the gate — an irrelevant group's subtree never emits, so the case block simply doesn't happen (m2-f0's client create is gated by `<bind nodeset="…/client_profile_save_to_case" relevant="/data/all_remaining_referral_case_ids != ''"/>`). Read any block count in this memo accordingly: a "20-block" form fires only the subset whose enclosing groups are relevant on a given submission.

### 1.4 Where processing happens, and the constraint that follows

On Web Apps, formplayer processes the submission's case blocks into the user's local sandbox at submit time (`formplayer/src/main/java/org/commcare/formplayer/application/FormSubmissionHelper.java` → `FormRecordProcessorHelper.processXML`, followed by a purge pass), and HQ processes the same blocks authoritatively on receipt. The client parser **errors if an `update`/`close` targets a case absent from the local sandbox** (`CaseXmlParser.loadCase`: "Unable to update or close case X, it wasn't found"). Consequence: ACA does not magically grant access to arbitrary ids — a form that updates case X must arrange for X to be locally present first (own it, have it ride in via an index chain, or claim it on the way into the form). The two apps' aggressive claim-on-entry configuration (§2.4) exists substantially to satisfy this.

### 1.5 How HQ even knows the form touches cases

HQ's app model does not parse ACA blocks out of the XForm. Instead Vellum reports a summary side-channel at save time (`saveToCase.js::getCaseSaveData` → the form's `case_references` — `commcare-hq/corehq/apps/app_manager/models.py::CaseReferences`, whose docstring admits it "is dependent on Vellum's API that sends case references to HQ"). That summary feeds the app's Case Summary and the data dictionary (`app_schemas/app_case_metadata.py::_add_save_to_case_questions` — which *skips the question entirely if no case type was set*; `app_manager/tasks.py::_refresh_data_dictionary_from_app`, gated on the paid privilege). Everything downstream that "knows about" ACA knows only what Vellum chose to report.

## 2. Why it exists: ownership, restores, and search-and-claim

ACA's capabilities sound like an arbitrary grab-bag until you hold the platform's data-visibility model in your head. Build it in four layers.

### 2.1 Ownership is visibility

Every case has an `owner_id` — a user, a case-sharing group, or a **location** (an organization-tree node). A user's "owner ids" are their own id plus their case-sharing groups, with owning locations contributing group-objects (`commcare-hq/corehq/apps/users/models.py::CouchUser.get_owner_ids`; `corehq/apps/locations/models.py::SQLLocation.case_sharing_group_object`). Location *types* carry the governing flags (`locations/models.py::LocationType`): **Owns Cases** (`shares_cases`) — cases owned at this level sync to users assigned here; **View Child Data** (`view_descendants`, bounded by `expand_view_child_data_to`) — users here also receive cases owned by descendant locations; **Has Users**; and a set of *location-fixture* scoping fields (`expand_from`, `expand_to`, `include_without_expanding`, `include_only`) that control which locations a user can *reference* in expressions, independent of which cases they receive.

The Colorado project space (from its Organization Levels & Structure configuration in HQ, July 2026) is a worked example:

```
State (has users, owns nothing)
└─ Registry level (owns cases, NO users)         ← "Client Registry", "Facility Registry", "BHASO Registry"
   └─ Organization (owns, has users, View Child Data → Facility)
      └─ Facility (owns, has users)              ← a clinic; workers assigned here
         └─ Facility Data (owns, NO users)       ← "<facility>_data" bucket
```

How the levels are actually used (builder-confirmed July 2026, plus form evidence):

- **Client Registry: the PHI vault nobody syncs.** Client cases are owned there; no user is assigned there, so no one restores them — they are reachable only through case search (§2.4). The privacy design concentrates here too: **only the client case carries PHI**. Everything else references clients through a generated "friendly id" (an adjective-noun-hash built at registration — Create Profile and Refer (m2-f0)'s setvalues), and Client Care Search stored no client name or DOB at all until the pending-admissions work arrived (visible in the census: m2-f0's client create writes no name/DOB properties; Add Client to Pending Admissions List (m15-f0)'s does).
- **Facility: where users live, and their standing directory.** Workers are assigned here, and the facility's own `clinic` case is owned here — so every worker permanently restores their site's directory record (which is why Update Bed Availability (m3-f0) selects clinics straight from `casedb` with no search).
- **Facility Data: the slim-restore bucket.** Workflow records — referrals, units, capacities, websites, edit requests — are owned by the facility's `_data` child, where *no users are assigned*, precisely so they do **not** ride in every facility worker's restore (the builders' stated primary driver for the level's existence). Workers reach them through search and claim instead. Privacy is a secondary, project-dependent benefit — largely moot here given the PHI-only-on-client rule. One deliberate exception proves ownership level is chosen per-workflow: pending-admissions-list referrals are owned at the **Facility** level (m15-f0's owner bind copies the clinic case's own `@owner_id`), because that standing list is worked daily from the restore — Manage Client Details (m16-f0) selects them from `casedb`, not from search results.
- **BHASO Registry: a bolt-on user-holder.** Added recently with the pending-admissions work so regional admins can see their region's data; the builders note they would likely have nested it into the main tree had it been a day-one requirement.
- **State: owns nothing.** Statewide staff (registry and care-coordination roles, gated by user-data flags rather than by location) work entirely from search queues. Escalated referrals, for instance, stay owned by the *originating* facility's bucket (Outgoing Referral Details (m6-f2)'s `contact_support_group/owner_id`, `'-'` for unknown referrers) and are found by `current_status = 'escalated'` searches, never by ownership.

Addressing is what makes any of this writable. Three distinct "address books" appear in the apps' owner-id expressions:

1. **The locations fixture** — pushed wide by *Include without expanding = Registry* on Organization and Facility ("include all levels of this type and their ancestors" — `LocationType.include_without_expanding` field comment), so every worker can look up locations they will never own at: `instance('locations')/locations/location[@type = 'registry'][name = 'Client Registry']/@id` (m2-f0). Custom location data fields surface as fixture *attributes*, providing join keys: every facility_data location carries `@facility_id` = its parent facility's id, so the recurring idiom `…location[@type = 'facility_data'][@facility_id = <a clinic case's @owner_id>]/@id` hops from any clinic case to that facility's bucket (Manage Provider Location (m4-f0) and everywhere else).
2. **Mobile-report (UCR) fixtures** — ambient data pipes (§4.6): the "Clinics by Case ID" report maps *any* clinic case, far beyond the user's casedb, to its facility, and sits in the middle of owner-resolution expressions (m2-f0's referral owner; `service_owner_id` in Central Registry's Admit Client (m6-f0)).
3. **Lookup tables** — `item-list:bhaso_regions` maps a region name to that region location's id for region-addressed messages (m15-f0).

The fixtures are the address book; ownership is the mailbox.

Default ownership, for contrast: with case sharing off, a created case is owned by its creator; with it on, the user must resolve to *exactly one* sharing group or the form errors out ("Case Sharing: Locations vs Case Sharing Groups", GS — a frank internal truth table of the error cases). Multi-location workers make the defaults a minefield, which is one quiet reason these apps set `owner_id` explicitly on essentially every create.

### 2.2 The restore is ownership plus graph closure

A sync/restore delivers: all open cases owned by the user's owner ids, then a closure over the index graph (`commcare-hq/corehq/ex-submodules/casexml/apps/phone/data_providers/case/livequery.py::do_livequery`; mirrored on-device by `commcare-core/.../cases/util/CasePurgeFilter.java`). The rules that matter, per the livequery docstring's worked examples and the "Extension Cases" (GS) sync table:

- **Child → parent**: a synced case pulls its parents (even closed ones). Parents do *not* pull children.
- **Extension ⇆ host**: an owned extension pulls its host *and vice versa* — a host pulls its open extensions, recursively, in both directions along chains.
- **Death**: closing a host abandons its extensions (`a(closed) ←ext— d(owned) ⇒ both purged`); an extension is never on a device without its host.
- Extension cases are conventionally **unowned**: created with `owner_id = '-'` (`UNOWNED_EXTENSION_OWNER_ID`, `casexml/apps/case/const.py`; HQ's own subcase generator writes it, `app_manager/xform.py`). They exist to ride along with their host.

So the restore is not a list; it is a *territory* — everything reachable from what you own, under extension-biased traversal rules. Owning one case can pull an entire tree ("Case Claim", GS: "once you've claimed and own one … the entire tree of cases is synced").

### 2.3 Restore size is the performance budget

"The restore file is one of the biggest predictors of how an application will perform for a user" ("USS Designing and Building Performant Apps", USH — the internal playbook). Every calculation in every form evaluates against it; public guidance caps comfortable caseloads around 1,000 (~25 cases ≈ +1s sync — "Managing Application Size as it Relates to Performance", commcarepublic).

Web Apps sharpens this dramatically. Both Colorado apps set the custom property **`cc-sync-after-form: yes`** (visible in each `profile.ccpr` under "Custom Properties added in HQ"). It makes formplayer sync after *every* form submission, or every five minutes, overriding the app's Auto-Sync Frequency ("CommCare Developer Options - Internal", GS; implementation `formplayer/.../util/FormplayerPropertyManager.java::isSyncAfterFormEnabled` → `FormSubmissionHelper::performSync`). Notably this is not even a first-class HQ setting — on the HQ side it exists only as a hardcoded key in the Web Apps frontend (`corehq/apps/cloudcare/static/cloudcare/js/formplayer/constants.js::POST_FORM_SYNC`), reachable through the free-form "custom properties" flag. With a sync on every submission, restore size stops being a login-time cost and becomes a *per-form* tax. The playbook's conclusion: "Only cases that users always need to access should be in the restore file."

### 2.4 Search and claim: access without ownership

The escape valve is **case search** — a live Elasticsearch query from the case list, crossing ownership boundaries entirely (`<query>`/`<remote-request>` in the suite; "Case Search and Claim", GS) — plus **claim**: selecting an un-owned result POSTs to `/phone/claim-case/`, and HQ manufactures an *extension case* of type `commcare-case-claim`, owned by the user, host-indexed to the found case (`commcare-hq/corehq/apps/ota/views.py::claim`; `casexml/apps/case/cleanup.py::claim_case`; `CLAIM_CASE_TYPE`, `case_search/models.py`). By §2.2's rules, that little case drags the host — and the host's whole extension tree — into the user's next restore. Closing the claim case releases it all. Formplayer auto-fires the claim when a search result isn't in the sandbox (`formplayer/.../services/MenuSessionRunnerService.java::doPostAndSync`; HQ answers 204 for "already claimed").

The Colorado apps run this at maximum intensity. Nearly every entry in both suites carries a claim `<post>` (`post-data case_id=…` guarded by `count(instance('casedb')/casedb/case[@case_id=…]) = 0`); case lists are built over live `results` / `results:inline` instances rather than the local casedb; `cc-index-case-search-results: yes` is set to keep those result instances fast. And the project pairs it with an **Automatic Case Update Rule** (the "Automatically Update Cases" machinery, commcarepublic; 50k updates/day cap) that *closes every `commcare-case-claim` case nightly* — domain-level configuration per team context (it lives outside the `.ccz`), and precisely the playbook's advice ("regularly close the commcare-case-claim cases … prevent a user's restore growing over time"). Each morning every worker's territory has collapsed back to nearly nothing, and the day's claims rebuild only what they actually touch. The system is self-healing by design.

Search performance is its own war story (builder-provided). The monster concatenated `_xpath_query` filters visible throughout the suite are deliberate: the searches began as many separate `ancestor-exists`/`subcase-exists` clauses whose load times were severe, and were merged into the single combined queries that exist now. The project ultimately received **dedicated case-search Elasticsearch capacity** — it is literally the documented example in HQ's own settings (`settings.py::CASE_SEARCH_SUB_INDICES`, whose comment reads `'co-carecoordination-perf': {'index_cname': 'case_search_bha', …}`) — plus, at the builders' urging, response caching on the sync/claim side. A pointed contrast for Nova, whose case store is Postgres with real relational links rather than search-index-over-document-blobs.

### 2.5 The punchline

Now the grab-bag becomes a coherent instrument. In this architecture:

1. **The session cannot carry what forms must write.** Forms are entered from search results and claim-loaded cases; the things a workflow must create or update are routinely not selectable — they don't exist yet, or belong to a location the user will never own at, or were located by id inside the form itself.
2. **Ownership is an address.** Creating a case owned by the Client Registry *is* publishing it to the registry; creating a case owned by another facility's data location *is* delivering it there; creating an extension case owned by yourself *is* subscribing to its host. Every one of those is a write to `owner_id` on a case the session never selected.
3. **The graph is infrastructure.** Extension indexes decide what rides along in whose restore; adding and removing them is how workflows bundle, hand off, and release data.

Ordinary case management binds every write to the session's selected case and its immediate children. ACA binds writes to *data* — any id an expression can produce, any owner the fixture can name, any multiplicity a repeat can enumerate. The public docs compress this into one sentence worth quoting verbatim: **"save to case allows you to make changes to cases that are not currently open in the form"** ("Enabling Custom Case Management Workflows with Save to Case", commcarepublic). The rest of this memo is what that sentence turns out to mean in practice.

## 3. When it's the right tool: the patterns the two apps actually use

Both apps use ordinary case management where it suffices — followup forms carry a plain, often *empty* `/data/case` block tying them to the session case; usercase writes use the standard config (`/data/commcare_usercase/case`); a few hidden modules use standard registration/subcase blocks. Everything else — the actual business logic — is ACA. A census of the `.ccz` form XML finds case-transaction blocks in 29 of 31 Client Care Search forms and 25 of 26 Central Registry forms, the large majority hand-authored. Eight recurring patterns, each impossible or badly contorted under ordinary config:

**P1 — Create into someone else's territory (ownership as routing).**
Create Profile and Refer (m2-f0) creates the `client` case owned by the **Client Registry** location and, per selected facility, a `referral` case extension-indexed to the client and owned by the **referring facility's data bucket** — not the destination's. The owner bind is the addressing model in miniature: `instance('locations')/locations/location[@type = 'facility_data'][@facility_id = instance('commcare-reports:…')/rows/row[column_1 = <referring clinic case id>]/column_9]/@id` — a mobile-report row maps the clinic case to its facility, and the fixture's custom `@facility_id` attribute maps the facility to its bucket (§2.1); an unknown ("other", non-LADDERS) referrer falls back to `owner_id = '-'`. The *destination* facility deliberately never owns the referral: it gets a `message` case in its own bucket (P2) and finds incoming referrals by search (Incoming Referrals (m9)). Central Registry's Admit Client (m6-f0, m10-f1) runs the same geometry from the receiving side — client → registry, `service` → the *admitting* clinic's bucket (`service_owner_id`, same report+fixture two-hop). *Ordinary config cannot do this at all for the client/referral pair: a basic followup form can only open child cases of the session case, and here there is no session case — the form is entered from a multi-select of search results.*

**P2 — Notification cases (ownership as messaging).**
Both apps model notifications as `message` cases: created with `owner_id` = the *recipient's* bucket — the destination facility's data location (`destination_clinic_owner_id`, copied straight from the destination clinic case's own `@owner_id`) or a BHASO region's location (resolved from the `bhaso_regions` lookup table) — extension-indexed to the referral they concern, carrying `message_type` and display fields; e.g. the trio of "create_message_…" blocks in Respond to Referral Request (m6-f1) that notify the referring clinic of a rejection or an info request, or the pregnant-client alerts in Manage Client Details (m16-f0). Delivery is the domain's **Conditional Alerts** (CommCare messaging, configured in HQ, invisible in the `.ccz`; project configuration reviewed July 2026): rules fire on `message_type` and their recipient options are built on the same ownership model — "The Case's Owner" (owner-as-address, literally), "Email Specified via Case Property" (`destination_clinic_team_email` and friends), and a custom "the case owner location's *parent* location" recipient that solves the buckets-have-no-users problem by delivering to the Facility above the owning Facility Data node — with per-user delivery filtered against the usercase alert preferences set in Manage Email Settings (m31-f0) (`referral_rejected_alerts` etc.), wired into the alert as user-data JSON filters. Messages created with `owner_id = '-'` are the other delivery mode: unowned, riding their host referral's sync, reaching people by email-property recipients or by already having the host.

**P3 — Moderated writes (requests instead of edits).**
Central Registry's clinic-side users may not edit registry-owned client/service records directly. Update Client Information Request (m0-f1) and Update Lock Status Request (m0-f3) instead *create* an `edit_request` case — owned by the **requesting facility's data bucket** (the standard two-hop lookup), extension-indexed to the target client/service, carrying the proposed field values as properties — and immediately self-claimed (P4), which is what actually keeps the request visible to its creator: bucket ownership alone puts it in no one's restore. Registry staff work a search-driven queue (Pending Requests (m15/m18), whose `x_commcare_custom_related_case_property='request_root_id'` pulls each request's root case into the results alongside it — `case_search/utils.py::get_expanded_case_results`) and, in Respond to Update Admission Request (m15-f2) / Respond to Pending Request (m17-f0), approving *applies* the requested values to the real cases via ACA update blocks; rejecting stamps the request. This is a four-eyes authorization workflow built from nothing but ownership boundaries plus ACA — the platform has no native concept of it.

**P4 — Hand-built claims (forcing the sync graph).**
The apps create `commcare-case-claim` cases *themselves* whenever a workflow's author knows the user will need a case in their restore that they just created into someone else's territory: `create_first_referral_case_claim` in Create Profile and Refer (m2-f0), `create_case_claim` after every client/edit-request creation in Central Registry (m0-f1, m0-f3, m6-f0, m10-f0/f1, m15-f0, m18-f1). The blocks replicate `cleanup.py::claim_case`'s exact shape — type and name `commcare-case-claim`, owner = the user, extension index → host. Unofficial (nothing validates client-manufactured claim cases; the endpoint's dedupe is bypassed) but wire-legal, and the nightly auto-close rule garbage-collects them indistinguishably. Duplication is avoided structurally rather than by the endpoint: hand-built claims only target cases created *in the same submission* (no prior claim can exist), and entry-level claim posts are presence-guarded (`relevant="count(instance('casedb')…) = 0"`). This is the team's known-but-unsupported trick — manually creating claim cases to force particular cases into the restore — confirmed in the wild.

**P5 — Fan-out maintenance of denormalized data.**
Case lists and search results can only cheaply display properties on the listed case itself, so the apps copy display fields across the graph — and must keep the copies fresh. Update Bed Availability (m3-f0) walks a clinic→unit→capacity repeat tree updating `clinic_availability_*` mirrors on every descendant; Manage Unit (m5-f0) recomputes `capacity_table_complete_display` (a markdown table stored as a property!) and clinic-level copies; Manage Provider Location (m4-f0) pushes clinic field changes down to all capacity children. The internal performance playbook documents this exact practice, *citing this exact app family* ("USS Designing and Building Performant Apps", denormalization section: the Colorado clinic/unit/capacity examples). One ACA block inside a repeat = one update per iterated case; ordinary config has no update-N-existing-cases primitive at all.

Ground truth on intent, from the builders: denormalization here is *imposed, not desired*. Web Apps case tiles render one case per row, or rows grouped under a shared parent header; to get a clinic header with per-bed-type rows beneath it, the case list must be `capacity` cases tile-grouped by their clinic parent — the list began life as a unit list and moved to capacity for exactly this reason — and every clinic field the tile shows must therefore live *on the capacity case*. The display layer's constraint is paid for in ACA maintenance blocks ("if it makes the design work… it is what it is"). A corollary for §6: an authoring system whose case list could project parent fields at display time would remove the *reason* for much of this pattern — though any app exported to CommCare still inherits the constraint on the wire, so the copies can at best be generated and maintained mechanically rather than by hand.

**P6 — Close as delete, close by id.**
"Delete" UX is a close block against an id computed in a repeat over existing records: websites/documents in Manage Provider Location (m4-f0), template units in Create New Units (m24-f0), lock releases in Update Lock Status (m0-f2) and Discharge Client (m0-f4). Ordinary config can close only the session's case.

The inverse rule matters just as much: **the apps almost never hard-close their primary entities.** Clients and referrals are "closed" by setting `current_status` while the case stays open, and every search and list filters on that property. The builders' reason: a hard-closed case is beyond the app's reach forever — nothing inside CommCare can reopen it — and the code adds two sharper edges: case search excludes closed cases outright (`case_search/utils.py` applies `.is_closed(False)` to every query; the module-level `include_closed` flag is marked deprecated in `case_search/models.py`), and a claim's extension dies with a closed host (§2.2). A hard-closed case can be neither found nor durably claimed from inside the app. So hard close is reserved for records that are *finished as data* — released locks, superseded duplicates, template scaffolding, the nightly claim sweep — where vanishing is the point.

**P7 — Graph surgery: link, re-link, unlink.**
The starkest ACA-only pattern. Request Admission Review (m10-f0) *indexes potential-duplicate client cases as extensions of the new `pending_admission`* — so whoever claims the pending admission automatically receives every candidate match in their restore, bundled for side-by-side review (a host pulls its open extensions, §2.2); Accept/Reject Pending Admission (m15-f0, 20 blocks) then **breaks** those temporary indexes by writing empty index targets (`break_index_between_clients_and_pending_admission…`, exploiting `CaseXmlParser.indexCase`'s empty-value-removes rule) *before* closing the pending admission — necessarily so, because closing a host cascades closure to its extension cases (`casexml/apps/case/xform.py::close_extension_cases`), and these "extensions" are real registry clients that must survive the review case. Resolve Duplicates (m16-f0, 16 blocks; m20-f0) is a full entity-resolution engine: create the merged client, re-point every service/lock/alias from the losers to the survivor via index-only blocks, convert loser clients into `alias` extension records, discharge conflicting services, close the losers with a `duplicate_primary_case_id` breadcrumb. (This is the workflow that got so large the team built a spreadsheet pipeline to generate the blocks — §5.)

**P8 — Multiplicity and conditionality from data.**
Throughout: one authored block, N effects (per repeat instance, per multi-selected search result), each block or property guarded by its own `relevant`. Create Profile and Refer (m2-f0) turns one form submission into 1 client + N referrals + N+ messages + 1 claim + a usercase update. Ordinary config's only multiplicity is child-subcases-per-repeat of the session case (`OpenSubCaseAction.repeat_context`, `app_manager/models.py`).

### 3.1 When it is *not* the tool

- **The session case's own lifecycle.** Register/update/close the selected case, child cases of it (even one per repeat), usercase properties — ordinary config generates the same wire blocks with none of the footguns, and it remains what these apps use for exactly that.
- **Session *reading* problems.** If the difficulty is which cases the user can *select* or which non-case data enters the session, that's the other escape hatch — Advanced Modules, which expose suite-level session datums. Dimagi's own guidance orders them: the Advanced Modules page opens with "consider whether you can solve the same problem without Advanced Modules (e.g. with Save to Case)" and later, bluntly, "If your primary use case for Advanced Modules is to create or modify cases that you wouldn't normally be able to access via Basic Modules, consider Save to Case instead" ("Advanced Modules", GS). Read: session shape → advanced modules; write shape → ACA. The Colorado apps use both, plus stack ops and endpoints, as one system.
- **When the target can't be guaranteed locally present.** An update/close against a case that won't be in the sandbox fails the submission (§1.4). If you can't claim it on the way in or ride an index chain, redesign (this is what P3's request pattern is *for* — write what you own, let the owner apply it).

**A note on form granularity.** The census's giant forms (Accept/Reject Pending Admission (m15-f0), Outgoing Referral Details (m6-f2)) are not a taste for monoliths; they are the path of least resistance on a platform where *chaining* forms is fragile (builder-confirmed). End-of-form navigation can launch a next form, but there is no reliable way to gate "step 2 of a flow": a stage flag has no guaranteed cleanup (a closed tab strands the user mid-flow until a nightly rule resets it), and the session stack fights back-navigation — the builders prototyped a timestamp-gated handoff (valid for ~five seconds after redirect) and never shipped it. So multi-step workflows collapse into one form whose conditional groups (§1.3) carry the combinational logic — and, under `cc-sync-after-form`, one submission also means exactly one sync. The builders are explicit that this is a compromise, not a preference (the client-edit section inside Outgoing Referral Details (m6-f2) is admitted accretion), which is why it matters for §6: form granularity in these apps encodes platform constraints, not workflow shape.

A useful compression of everything above — ordinary case management answers "what happens to the case I picked?"; ACA answers **"what does this *event* do to the *system*?"** A referral event touches a client in the registry, a referral in your own facility's data bucket (which still isn't in anyone's restore), messages in other facilities' buckets, a claim in your own pocket, and bookkeeping on the clinic — one submission, seven-plus cases, five owners.

## 4. Constraints, limitations, and footguns

Documented warnings first, then what the apps visibly fight.

1. **Gated, trained, and paid.** ACA requires an Advanced-or-higher plan *plus* mandatory "Deep Dive+" onboarding before Dimagi will flip the flag, explicitly because of "the complexity of this feature and the support burden of tickets" ("[CommCare Supported Feature] Save to Case Feature Flag", saas; privilege `VELLUM_SAVE_TO_CASE`, toggles `save_to_case` / `saas_formbuilder_save_to_case`).
2. **The raw-XPath targeting model has no referential integrity.** `case_id` is a free string expression. Vellum validates only its *shape* (expression-ness; `uuid()` only with Create — `saveToCase.js::case_id.validationFunc`). A wrong expression mutates the wrong case or errors at submit; nothing at build time knows whether the id will exist, be the right type, or be present locally.
3. **Local-presence requirement** (§1.4): update/close of a case absent from the sandbox fails the submission client-side. The apps' answer — claim-on-entry everywhere — costs a server round-trip and a sync per entry.
4. **Created-elsewhere data vanishes on the next sync.** A case created with a foreign owner and no anchoring index/claim is purged from the creator's device at the next sync (§2.2 — correct behavior, startling UX: "the referral I just made disappeared"). Hence P4's hand-built claims.
5. **Search lag.** Case search runs on Elasticsearch; a just-created case may not be findable for a beat. The apps hedge with `case_search_ts` stamps, related-case pulls (`x_commcare_include_all_related_cases`), and exact-id hydration via the `phone/case_fixture/` endpoint in smart-link stacks; a domain setting for synchronous ES indexing of Web Apps submissions exists ("Synchronous Web Apps Submissions to ElasticSearch", USH).
6. **HQ is nearly blind to it.** Case Summary/data dictionary knowledge comes from Vellum's summary side-channel and *silently omits* questions with no static case type (§1.5). Form exports, case-type graphs, and "where is this property written?" tooling under-report ACA. Relatedly, both apps carry a population of hidden modules that smuggle in capabilities the platform doesn't offer directly (builder-confirmed): an **icon-holder case list** ("[Hidden] Icon List" (m26) in Client Care Search, "Icon Holder" (m28) in Central Registry) exists because CommCare has no way to upload arbitrary media — a case list's icon columns are the upload surface, and the assets are then referenced from markdown everywhere else; **shadow modules** re-expose one form under several menus without duplicating it (why Respond to Referral Request appears as both m6-f1 and m9-f1 over the same `modules-6/forms-1.xml`); and **mobile-report modules** exist not to show reports but as *data pipes* — the "Clinics by Case ID" UCR gives every form a table of all facilities regardless of ownership, powering dropdowns and even the owner-resolution expressions of §3-P1. A last few stub forms with junk bindings (Register (m3-f0), the relationship forms (m25-f0, m26-f0)) remain unexplained — see §7.
7. **The archetype straitjacket forces stubs.** HQ still requires, e.g., a registration form to have its generated open-case block, so the apps *disable* it with never-true conditions — `<bind nodeset="/data/case" relevant="/data/open_case = '2'"/>` where `open_case` is an empty node (Respond to Pending Request (m17-f0); same pattern with `avoid_automatic_reg` / `prevent_automatic_case_mgmt_reg` in Create New Units (m24-f0), Create Location (m29-f0), Create Non LADDERS Organization (m30-f0)) — and rebuild creation in ACA to control owner, id, and multiplicity. The generated stub's leftovers are nonsense on purpose (case_name bound to whatever question was handy). The same straitjacket explains the last mystery modules: **HQ requires every case-list module to contain at least one form**, and the default flow (create case list → HQ scaffolds a followup form → delete it → error → put it back) is why Central Registry's junk-binding forms (Register (m3-f0), the relationship forms (m4-f0, m25-f0, m26-f0)) exist at all — filler to satisfy the requirement, with whichever case config HQ generated by default (which also dissolves the child-vs-extension "inconsistency": the stubs' `child` subcases were never a choice; the live paths' `extension` everywhere was, for the sync mechanics). A case-list menu *item* quietly exempts a module from the form requirement, but that is documented nowhere and the builders only discovered it later — during Nova's own development.
8. **Authoring does not scale in the UI.** The Colorado teams maintain a Google-Sheets pipeline that *generates* Save to Case configuration for copy-paste into the form builder because typing 200–300 property rows is unworkable — created for a sibling Colorado program's deduplication merge forms (CO CICT) and adopted by this app family's team (a CO BHA builder reports "upwards of 300 case properties … in a single save to case question"), with a warning that "single versus double quotes matter!… be ready to potentially deal with many tiny formatting errors" ("Building Big Save to Cases in Excel/Sheets", USH). The census's 20-block forms are the same pressure from the block-count direction: case effects smeared across a form tree with no unified view.
9. **Repeat-group paths are subtle.** Blocks inside repeats address per-iteration values with `current()/../../../../…` relative paths (e.g. `@case_id calc: current()/../../../../case_id` throughout Accept/Reject Pending Admission (m15-f0)); an internal worked example exists solely to teach this ("Nested Repeat Groups Example", GS). In-repeat creates also switch `@case_id` from setvalue to bind mechanics (`saveToCase.js::getBindList` vs `getSetValues`) — invisible in the UI, meaningful on the wire.
10. **Special case types have special rules.** `commcare-user` (the usercase) may only be *updated*; `user-owner-mapping-case` is reserved (`saveToCase.js::case_type.validationFunc`). Nothing stops you from creating `commcare-case-claim` cases — the apps depend on that — but it is an undocumented, unvalidated workflow.
11. **Wire-vs-UI drift.** The wire happily accepts create+update in one block, arbitrary index identifiers, attachments, and owner-transfer via update; the new UI constrains some of these (exclusive create/update chips, no attachments section). Old apps (these two included, pre-rework) carry shapes the new UI normalizes on load (`legacy_case_type_bind.xml` handling, relevants promoted to conditions).
12. **Old workarounds calcify.** The unit search carries an always-true filter, `concat('@case_id != "', uuid(6), '"')` — a relic from before CSQL had `match-none()` (it does now: `case_search/xpath_functions/__init__.py`), when a query that matched nothing errored. The hack outlived its reason and nobody remembers exactly why it survives (builder-confirmed); hand-authored wire hacks have no expiry mechanism.

## 5. How HQ presents it (Vellum), and an honest assessment

**The presentation.** ACA is a *question type*. You add "Advanced Case Actions" from the Advanced submenu wherever the cursor happens to be in the form tree; it renders as a data node with a diagram icon. Its properties panel (post-2026-rework) is genuinely well-organized for a single block: action chips with tooltips up top; Case Type from a data-dictionary dropdown; Case ID as an XPath box; collapsible Create/Update/Close/Link-Unlink sections that expand when their chip is on; property cards with name/calculation/condition; `date_modified`/`user_id` tucked under Advanced. The pre-rework UI — the one all documentation in the Confluence dump describes, and the one this feature spent its first decade as — was the same model minus the affordances: flat property lists, a hamburger menu to reveal hidden sections ("Save to Case Feature Flag" page's walkthrough), free-text case types, no chips. The Excel-pipeline page shows what building at scale against it was like.

**Where it helps understanding.** The rework makes the *anatomy* legible: the chips map one-to-one to wire actions, "Link / Unlink" is a better name for `index` than `index` ever was, per-property conditions are visible instead of buried in binds, and pulling case types from the data dictionary plus reporting writes back to it (§1.5) is real, current investment — Dimagi clearly considers the feature strategic rather than legacy (rework landed April–June 2026; the help page is newer than the Confluence dump).

**Where it hurts.**

- **It presents a system-level construct at question-level scope.** Each block is configured alone, in a properties panel, at some arbitrary node in the form tree. A form like Accept/Reject Pending Admission (m15-f0) is *twenty* of these scattered through groups and repeats; no screen shows "what does submitting this form do to the case universe?" — the closest thing (App Summary) sees only the Vellum side-channel. The mental model the *author* needs (§2's territory/address/graph model) appears nowhere in the surface.
- **Placement-as-configuration is invisible semantics.** Dropping the question inside a repeat changes it from "one case" to "N cases" — with different `@case_id` wiring — with zero visual acknowledgment.
- **The targeting model is a string.** Everything hard about ACA concentrates in the Case ID box, and the box is a bare XPath field. Wrong-id, stale-id, not-locally-present, wrong-type — all invisible until runtime.
- **Ownership — the entire *point* of the feature (§2) — is one optional "Owner ID" XPath field** with no awareness of locations, fixtures, buckets, or what the id will *mean* for visibility. The docs' own summary of `owner_id` ("only required when the ownership of the cases don't follow CommCare's default rules") describes the mechanism while completely missing the reason anyone uses it.
- **Residual incoherences.** Create and Update are exclusive chips though the wire (and old forms) combine them, and a create's properties live in cards that emit as `<update>` — the UI's names and the wire's names shear past each other; attachments exist on the wire but not in the UI; conditions are scattered (open-case condition, owner condition, close condition, per-property conditions) rather than unified.

Net: the 2026 Vellum is a competent *form* for filling in a case block. It is not an *explanation* of the feature, and it cannot be — the feature's meaning lives at the app-and-org level (who owns what, what syncs where), a level the form builder has no view of. That gap between where ACA is configured and where it makes sense is, more than anything else, why it costs Dimagi mandatory trainings and a support queue.

## 6. Preliminary implications for Nova (hypotheses for discussion)

Framing per house rules: the wire format is the only inheritance; Vellum's authoring shapes are anti-patterns to reject unless they'd be right built from scratch. These are hypotheses to argue about, not designs.

**H1 — Split the feature's uses into two tiers; they have different prerequisites.**
The two apps show ACA serving (a) *session-independent case logic* — fan-out updates in repeats, close-by-id, side-effect record creation (audit trails like `contact_attempt`, message-like records), graph link/unlink, event-shaped forms touching several known cases — and (b) *ownership-and-sync choreography* — foreign-owner routing, claim-forcing, moderated-write queues. Tier (a) is meaningful in any Nova app **today**: it needs only expressions that produce case ids, and Nova's preview (real Postgres rows, `lib/case-store`) can execute it faithfully. Tier (b) is only meaningful once Nova models locations/owners and search-and-claim — `owner_id` routing exports fine as wire, but the preview cannot *demonstrate* "this lands in another worker's restore" without an ownership model, and a feature whose core value can't be previewed is a UX lie by omission. **Decision to surface: does ACA enter Nova as a whole, or does tier (a) ship with tier (b) explicitly staged behind an ownership/search model?** Builder context: the project's own sequencing instinct is ACA before a locations model — which tier (a) supports, provided owner-id stays a wire-complete expression slot so tier (b) is emission-ready when the ownership model lands (see the companion locations memo, `commcare-locations.md`).

**H2 — Nova's shape should be "operations on cases," not "a question that is a case block."**
Vellum's question-node model (placement-as-multiplicity, config smeared across the tree, name-keyed `{calculate, relevant}` maps of raw XPath) is exactly the CCHQ-shape leak Nova rejects. A from-scratch shape: a form carries a list of **case operations** — each `{ action: create|update|close, target, caseType (catalog ref), writes: [(property ref, expression, condition?)], links: [...], condition?, forEach? }` — where `target` and every expression are Nova's typed ASTs, property references are `(caseType, property)` identities, and multiplicity (`forEach` over a repeat/collection) is an explicit field rather than a placement accident. Nova's reference model then delivers what Vellum structurally cannot: renames that don't rot expressions, and real referential knowledge of which types/properties a form writes (no §1.5-style side-channel — the doc itself knows). The emitter compiles operations to case blocks + binds (`lib/commcare`), which is mechanical. **Decisions: is `index` a first-class action or a `links` facet of create/update (the wire treats it as an action; authors think "this referral belongs to this client" at create time)? Is close-with-final-writes one op or two? Where do ops live — per-form list vs attached to repeats they iterate?**

**H3 — Valid-by-construction meets its first genuinely dynamic boundary.**
Nova currently guarantees an app can't be invalid; ACA targets are runtime-resolved by design, so "the target exists / is of the stated type / is locally present" cannot be a construction-time invariant — the strongest static guarantees available are catalog-level (case type exists; written properties belong to it; the target expression's *type* is a case-id of that case type, which Nova's typed ASTs could actually track) plus reachability heuristics (e.g. warn when an update's target can't be shown to be session-reachable/claimed — the §1.4 failure). Also: platform-reserved types (`commcare-case-claim`, the usercase) would need catalog representation to pass gating. **Decisions: how loud is the can't-prove-present warning; do reserved types enter the catalog as built-ins; does the case-store schema materialization treat ACA-written properties identically to module-written ones (it must — the preview writes real rows).**

**H4 — Name and narrate by intent, not mechanics.**
"Save to Case" describes the wire; the winning explanations in this research were all intent-level: *"make changes to cases that are not currently open in the form"*, ownership-as-address, requests-instead-of-edits. Nova's surface (and docs) should lead with the verbs — create/update/close/link *other* cases — and treat the CommCare vocabulary as export detail.

**H5 — For the SA, ACA is a set of recognizable workflow smells, not a feature to offer.**
Candidate trigger heuristics for agent guidance, distilled from §3 (each maps to a pattern):
- "When X happens, also update/mark/recalculate Y" where Y is not the case the form is about (P5, P8);
- "…and notify/alert/inbox the other team/site" (P2);
- "register into a shared registry / send to another facility / assign to a queue" (P1 — needs tier (b));
- "request/approve/review before it takes effect" (P3);
- "merge/deduplicate/re-assign records" (P7);
- "remove/archive items from a list" the form iterates (P6);
- one event, several records ("intake creates the client, the visit, and the referral") (P8).
Equally important negative guidance: if the mutation is the session case or its children, use ordinary actions; never reach for ACA to compensate for a missing selection structure (that's module/session design). And a calibration note from the builders themselves: these apps are "with limited resources, the best app builders at Dimagi tried their hardest to meet their partner's needs" — patterns to learn from, not artifacts to copy. AI authoring also changes the complexity budget that shaped them (a form that couldn't fit a sprint costs the SA minutes), so the agent should feel free to re-derive structure rather than inherit it — e.g. the builders themselves wonder whether `pending_admission` and `edit_request` needed to be distinct case types at all, versus a client `current_status`. **Decision: how much of §2's ownership model goes into the SA's standing prompt vs being introduced only when the app actually models multi-site ownership — the model is heavy, and most Nova apps today are single-territory.**

**H6 — The preview must execute, not simulate.**
Nova's flipbook runs the real engine on real rows; ACA ops must therefore run in `lib/preview` with the *same semantics the wire will have* — document-order application, empty-index-removes, close cascades (if/when Nova models extensions), create-overwrite tolerance — or the "no preview/production split" principle breaks exactly where users most need to see consequences. This is a real implementation cost to budget, not a rendering detail.

**H7 — There may be a Nova-native answer to the *stub* pathologies.**
Several CCHQ workarounds exist only because generated archetype blocks can't be turned off (never-true conditions, §4.7) and because HQ can't see ACA writes (§4.6's hidden-module menagerie). Nova, owning both the doc and the derivations, simply never needs these: forms are whatever their operations say they are, media is a first-class citizen, and reuse is a doc concern rather than a shadow-module hack. Worth stating as a design principle so the archetype vocabulary (registration/followup/close/survey) doesn't harden into the same straitjacket.

**H8 — Form granularity is a designed decision, not an inheritance.**
The mega-forms of §3 encode CommCare's flow-chaining fragility (no reliable step gating, no guaranteed cleanup, per-submission sync cost), not ideal workflow shape — the builders say so directly. Nova emits CommCare forms and cannot invent flows the runtime won't execute, but the SA should treat "one event, one form" versus "a chained flow" as an explicit choice made with knowledge of *why* CCHQ builders default to monoliths, and Nova's builder/preview can present a large event-form as structured steps even when the wire artifact stays monolithic. **Decision: whether Nova's form model gets first-class sections/steps that compile to conditional groups.**

**H9 — The app is not the whole system; the agent should design the domain-side pieces too.**
Two load-bearing components of these workflows live entirely outside the `.ccz`: the **automatic case update rules** (the nightly claim sweep that makes the self-healing caseload work, §2.4) and the **conditional alerts** (the delivery half of every message case, §3-P2). Neither has a public write API today — the builders are pushing for one and may PR CCHQ themselves — which means a Nova app that uses these patterns is *incomplete as shipped* unless Nova acknowledges the gap. The hypothesis: the SA should design rules and alerts as first-class outputs whenever a workflow depends on them (an owner-routed message pattern *implies* its alert; a claim-heavy design *implies* its sweep), with Nova emitting them as structured setup artifacts today (human-applied config, precisely specified) and pushing them via API the moment one exists. The alternative — the agent silently designing patterns whose critical halves nobody provisions — is the kind of vacuous success Nova exists to prevent. **Decisions: whether rules/alerts enter the blueprint vocabulary now (unexported but modeled) or wait for the API; and how the SA's guidance words the dependency so it plans for the ability without overpromising it.**

## 7. Open questions

1. **Index-only blocks against absent cases.** `CaseXmlParser.parse` routes `index` through `loadCase(errorIfMissing=false)` and then dereferences the result — which reads as an NPE if the case is missing locally, versus update/close's clean error. Does some guard exist upstream (bulk parser, formplayer), or is this a real crash edge? The failure *class* is confirmed live: the project's own error tracking shows recurring "unable to claim case"-family errors that the builders cannot diagnose, partly because app builders have no access to the technical layer that produced them (builder-reported). Worth a targeted test before Nova mimics semantics — and the diagnostic opacity is its own lesson for Nova's error surfaces.
2. **The auto-update-rule daily cap.** The public "Automatically Update Cases" page says up to 50,000 updates/day; the USS playbook says 10,000/domain/case-type. Immaterial at this project's volume (builder-confirmed: never hit), but worth resolving before Nova emits designs that lean on nightly sweeps at real scale.
3. **Multi-claim latency.** Formplayer claims every selected case on entry to a multi-select flow; the builders confirm it visibly drags at high selection counts in testing (each claim implies a sync under `cc-sync-after-form`) and accepted the cost rather than degrade the end-user workflow. No mitigation exists today; if Nova's designs use multi-select-and-claim, the latency curve deserves measurement.
4. **Attachments.** Wire-supported (`CaseXmlParser.processCaseAttachment`), absent from the new UI, unused by both apps — and unfamiliar even to a power builder (form attachments are the normal channel). Effectively dead in the field; the remaining question is only whether Nova's op vocabulary acknowledges it at all (probably not).

## Appendix: the two apps' data model

The program (builder-provided context, July 2026): this is the Colorado **Behavioral Health Administration**'s statewide substance-use-disorder care-coordination system — Client Care Search is the referral/bed-finding side, Central Registry the admissions ledger. The central registry, deduplication machinery, and admission locks all exist for the same clinical reason: preventing duplicate concurrent treatment across the state. "Lock In" binds a client to a clinic; "Lock Out" bars a facility. Facility directory data originates in **LADDERS** (the state's facility data system) and flows LADDERS → Snowflake → CommCare via API — which is why clinic/provider cases carry `data_source`/`external_id`, and why the "Non-LADDERS" creation modules (Create Location (m29), Create Non LADDERS Organization (m30)) exist for organizations outside that pipe.

The full case-type graph, with owners as verified in the form XML. Read the **owner** column as the workflow design (§2.5: ownership is an address) and the **closure** column against the soft-close doctrine (§3-P6):

|---|---|---|---|---|---|
| `client` | both | Client Registry location (fixture lookup by type+name) | — (the root others attach to) | **Yes — the only PHI carrier.** CCS held nothing but the friendly id until pending admissions; names/DOB/SSN live in CR | Soft (`current_status`); hard only as merge losers |
| `alias` | CR | Client Registry | → client | Name variants | Kept as records |
| `service` (admission) | CR | Admitting facility's data bucket (report+fixture two-hop) | → client (or pending_admission) | Minimal | Soft — discharge via `current_status`/dates |
| `pending_admission` | CR | Admitting facility's data bucket | — (candidate clients temporarily ext → *it*) | Intake fields incl. names | **Hard** once resolved (after unlinking candidates) |
