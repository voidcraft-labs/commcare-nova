**What ships.** The app-facing half in full: a typed **user-data schema** (the predicate
vocabulary F1's open-namespace `session-user` refs tighten into), first-class **user types**
(named, typed value-bundles that become the preview's personas and provisioning presets), and
the **usercase** as a built-in reserved case type with read *and* write authoring reusing
Nova's existing per-field mechanism. The provisioning half is planned and specced but
recommended to land with/after F3 — the user APIs are verified writable for *users and their
data values*, while the schema itself is **not API-pushable** and becomes a setup artifact.

## 2. The shape question (protocol 3)

**CCHQ's authoring shape** — a flat slug/label field list on a domain-wide definition, plus
paid profiles, plus stringly `user_data` dicts — is not inherited. **Nova's shape** is two
app-level collections plus a static built-ins catalog:

```ts
// blueprintDocSchema — two new collections (caseTypes-pattern clones)
userDataFields: z.array(userDataFieldSchema).nullable(),
userTypes: z.array(userTypeSchema).nullable(),

userDataFieldSchema = {
  name: string,       // the wire slug — fact 2's exact legality rule at construction
  label: string,
  data_type?: CasePropertyDataType,   // one declarable-type set repo-wide
  options?: Array<{ value, label }>,  // select-typed fields (maps to HQ `choices` at provisioning)
  required?: boolean,
}

userTypeSchema = {
  uuid: Uuid,          // stable identity
  name: string,        // "Facility worker", "Registry staff"
  description?: string,
  values: Record<string, string>,   // field name → value; checked against userDataFields
}
```

- **A "user type" is a first-class blueprint object** (charter Q1): the schema half
  (`userDataFields`) is the typed vocabulary; the types half (`userTypes`) is the named
  role-bundle — what the SA designs ("supervisors can approve"), what the preview
  impersonates, what provisioning presets. HQ's paid profiles are deliberately NOT the model
  (fact 3); a Nova user type compiles to plain per-user `user_data` values at provisioning.
- **Built-in user properties** are a static catalog, not doc content: the injected keys of
  fact 4 (`commcare_project`, `commcare_first_name`, …, location keys arriving with F3),
  typed and offered in pickers alongside authored fields. Reserved names (facts 2, 11) are
  unconstructible as authored fields.
- **User-data references tighten, not change**: F1's `session-user` Term is already the
  typed AST node; F2 adds the vocabulary check — a `session-user` field must be a declared
  `userDataFields` entry or a built-in. The gate is introduce-only (legacy open-namespace
  refs in existing docs keep validating; the new finding needs its repair judgment), and the
  checker's resolved type comes from the field's `data_type`. Rename-safety: field names are
  name-keyed identity (like case properties) with new reference-index edges + rename
  cascade arms over every `session-user` carrier.

**The usercase is a built-in reserved case type, not a new concept** (charter Q2). Evidence:
it IS a case end-to-end (facts 6, 10 — ordinary case XML, ordinary client processing;
`#user/` already resolves to it in Nova's XPath layer). So `commcare-user` enters the
case-type catalog as a non-removable built-in entry, seeded with its sync-provided built-in
properties (username, first_name, last_name, email, phone_number, commcare_project,
hq_user_id — fact 6) and extendable with app-authored properties. That representation is
exactly what F4's reserved-type gate (H3) wanted the catalog to carry. Authoring surfaces:

- **Read**: `#user/<prop>` XPath refs (existing leaf — now validated against the catalog
  entry's property set) and a new `usercase-prop` Term for the Predicate family (display
  conditions, filters, calculated columns) compiling to the verified
  `[@case_type='commcare-user'][hq_user_id = session/context/userid]` lookup on-device and
  to the persona's usercase row on Postgres.
- **Write**: a field with `case_property_on: 'commcare-user'` derives a usercase update —
  reusing Nova's one existing write mechanism, zero new field vocabulary. The derivation
  populates the **already-present inert FormActions slots** (`usercase_update` /
  `usercase_preload`), and the local path gains the missing `addCaseBlocks` usercase branch
  mirroring `_add_usercase` (verified absent today), plus the `usercase_id` computed datum +
  assertion in the suite (fact 7's shapes).
- Create/close of the usercase stay unconstructible (F4's reserved-type rule) — update-only,
  matching the platform's own authoring guard.
- **Export note, not a gate**: the usercase runs on HQ only with the paid `USERCASE`
  privilege (fact 6). Nova cannot see a target domain's plan, so authoring is ungated; the
  docs, the SA's guidance, and the HQ-upload surface carry the plan requirement explicitly.

## 3. Charter closure

### 3.1 References into the typed AST (charter Q1) — answered in §2.

### 3.2 Usercase modeling + preview (charter Q2)
Built-in catalog type per §2. **Preview**: yes — per-persona usercase rows in the case
store. Selecting a persona materializes (idempotently) a `commcare-user` case row keyed to
the persona's identity within `(app_id, project_id)`, owner = the persona; usercase reads
resolve against it and usercase writes update it — real rows, same store, no mock. The
persona's `session/user/data` values come from its user type's `values` + built-ins
synthesized from the persona (project name, persona name), with ad-hoc overrides preserved
from F1's substrate.

### 3.3 Provisioning (charter Q3) — verified surface, revised bet
What the APIs support (fact 8): create/update mobile workers with `user_data` values
(identity: `user_id`; username immutable), invite web users with role + user_data. What they
do NOT support: pushing the field-definition schema or profiles. Consequences:
- Nova pushes **users and their values**; the HQ-side schema definition ships as a **setup
  artifact** (precise, human-applied: field slugs/labels/choices to add under Users → User
  Fields) — the F6 output pattern arriving early, kept in sync with `userDataFields` at
  export time. Values work at runtime regardless of the definition (fact 9's verbatim
  parse); the definition is HQ-UI hygiene.
- **Sequencing revision (map bet, revised with reasons):** the provisioning prompts are
  specced here (P6) but recommended to land **with F3** — worker provisioning without
  location assignment is hollow for exactly the apps that need it (LOC §1: assignment is
  what turns users into caseloads), and the map already granted the halves independence.
  Nothing in P1–P5 depends on P6.

### 3.4 Project members vs app personas (charter Q4, jointly with LOC L5)
Three distinct populations, no blending:
- **Project members** (Better Auth organizations; flat viewer/editor/admin/owner in
  `lib/auth/projectRoles.ts`) are *authors* — Nova's collaboration/tenancy axis. They are
  never app users and never enter the blueprint.
- **User types / personas** are *design artifacts in the blueprint* — they parameterize the
  preview (F1's toggle grows into a typed persona picker) and describe intended runtime
  roles. L5's "preview-as-persona" question is answered here: personas are app-level
  objects, NOT Project members — an author previews *as a persona*, never as a teammate —
  and F3 extends the same persona objects with location assignment rather than inventing a
  second concept.
- **Provisioned workers** are HQ runtime users created FROM user types (P6), keyed by
  `user_id` per push.

## 4. Full-stack scope (protocol 4)

Domain: the two collections + built-ins catalog + slug rules + `usercase-prop` Term +
`session-user` vocabulary check + catalog built-in type + reference edges/cascades
(`session-user` carriers; `userType.values` keys; `#user` property refs). Doc/mutations:
keyed mutation families for both collections (declare/retire pattern; removal of a
referenced field blocks with references). Validator: new codes + classes + repair judgments
(unknown user-data field — introduce-gated; type errors; reserved names unconstructible;
usercase property checks; user-type value/type coherence). Emitters: usercase FormActions
population + `addCaseBlocks` branch + `usercase_id` datum + assertion + `#user`/
`usercase-prop` lowering; HQ JSON untouched otherwise. Preview: persona picker (userTypes) +
session-user resolution + per-persona usercase rows + usercase read/write in submissions.
Builder UI: a Users workspace (schema editor + user types editor), persona picker in the
preview shell, usercase-property affordances on fields. SA + MCP: tools for both collections
+ usercase params + guidance (role vocabulary with F1 display conditions; usercase for
per-user preferences/scratch — ACA §3-P2's alert toggles; negative: usercase is per-user,
never shared state; the HQ privilege notes). Docs: authoring page + privilege/plan notes +
setup-artifact explanation + tools.mdx + CLAUDE.md updates. Migration: none (additive);
F1's open-namespace refs remain valid by introduce-only gating.

## 6. Risks + notes

- **Paid gates** (`USERCASE`, `APP_USER_PROFILES`, regex enforcement): all authoring-side on
  HQ; Nova's exports carry plan-requirement notes rather than gating. If a target domain
  lacks `USERCASE`, HQ's build validation surfaces it at "Make New Version" — documented.
- **Schema-not-pushable** is the one hard API gap (fact 8); the setup artifact is the
  designed mitigation, and F6's planning should treat it as the same output family as
  rules/alerts.
- **Removed-key staleness** (fact 9's incremental-merge nuance) is a platform behavior Nova
  documents, not simulates.
- The demo-sandbox embedding (fact 12) is recorded as out-of-scope for the web-apps target;
  if a mobile-offline story ever matters, the mechanism is verified and waiting.
