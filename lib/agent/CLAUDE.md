# lib/agent ā€” LLM-facing agent layer

Owns every module the Solutions Architect reaches into during generation or edit. No other directory constructs an LLM provider (every model call rides the Vercel AI Gateway ā€” one `AI_GATEWAY_API_KEY`, gateway-format model ids from `lib/models.ts`), renders the SA system prompt, or generates tool schemas.

## Boundary rule

External consumers (`app/api/chat/route.ts`, `app/api/compile/route.ts`, `components/chat/ChatSidebar.tsx`) import from `@/lib/agent/*` entry points ā€” never from individual implementation files.

**The SA speaks domain vocabulary end-to-end.** Tool names, tool arguments, tool return shapes, and the system prompt all use domain names (`field`, `kind`, `validate`, `validate_msg`, `case_property_on`). There is no CommCareā†’domain translation layer anywhere in this directory ā€” SA tool args feed directly into `blueprintHelpers.ts` reducers.

The case-type pointer is `case_property_on` (disambiguating against the still-present `CasePropertyMapping.case_property` slot in `lib/domain/blueprint.ts`, which holds a property name). The `_on` suffix is load-bearing: it forces the prepositional reading "the case [type] this property is on," which keeps the SA from treating its value as a property name. Without the suffix the SA reads the value as a property name and prefixes field ids with the case-type name to "disambiguate," wasting tokens and corrupting field identity.

CommCare wire terms live at one genuine boundary outside `lib/agent/`: `lib/commcare/` (XForm emission, HQ JSON expander, validator, suite-entry derivation). The commit gate feeds `BlueprintDoc` through the validator directly ā€” no wire-format round-trip inside the agent layer.

## What lives here

- `solutionsArchitect.ts` ā€” the one `ToolLoopAgent` factory. Owns the SA's internal `BlueprintDoc` for the lifetime of a request and emits fine-grained mutations for every tool call. Tool execution is serialized via a promise-chain mutex so parallel `tool_use` blocks see a consistent working `doc` ā€” without it, concurrent branches each read the same pre-batch snapshot and the last `doc = newDoc` assignment silently drops earlier mutations from the SA's view (mutations still stream to the wire; only the SA's own state is corrupted, which surfaces as a wasteful "edits aren't sticking" rework loop). There is no finishing tool: the chat ROUTE finalizes a build at drain end ā€” it drains the run's chained saves, awaits `materializeCaseStoreSchemas` (UPSERTs `case_type_schemas` rows + per-property indexes), flips `generating ā†’ complete` WHILE settling the run's kept charge in one transaction (`completeAndSettleRun` ā€” the atomic build-completion writer; a clean EDIT completion is the parallel `clearRunLockAndSettle`, which releases the `run_lock` + settles atomically), and only THEN emits `data-done` ā€” so any user-initiated case-store action (sample-data populate, form submit, live preview) that fires sub-second after the celebration animation sees a synced Postgres schema. A throw out of any finalize step routes through the route's `failRun` (classify + emit + refund + `failApp` for a build). See `lib/db/CLAUDE.md` Ā§ finalization invariant for the full claim/settle/reap lifecycle.
- `prompts.ts` ā€” composes the mode-specific (build vs. edit) SA system prompt. CommCare XForm terms (e.g. `jr:`, `#form/`, XPath function names) still appear where the SA genuinely needs them.
- `summarizeBlueprint.ts` ā€” domain-vocabulary renderer that walks `BlueprintDoc` and produces the compact text both `prompts.ts` (edit-mode summary) and the MCP `get_app` tool share.
- `toolSchemaGenerator.ts` ā€” generates the two field-mutation tool inputs (`addFieldsItemSchema`, `editFieldUpdatesSchema`) from `fieldRegistry` + `fieldKinds`. Each is a `discriminatedUnion("kind", ā€¦)`: an arm exposes ONLY the properties that kind's domain schema declares (gated by `fieldKindDeclaresKey`) and is `.strict()`, so a property the kind doesn't have ā€” `calculate` on a `single_select`, `required` on a `hidden` ā€” isn't a slot on the arm and an explicit attempt is rejected at the tool boundary. That's the structural reason the wrong-property-for-kind error class can't be expressed. Each arm's `kind` literal carries that kind's `saDocs` as its description. Tool use is NOT grammar-constrained (that's `Output.object` only), so the arms carry as many optionals as the kind declares ā€” no ceiling. `label` is per kind on the ADD arms: omitted on `hidden`, optional on the containers (`group` / `repeat`), required + non-empty on every visible kind ā€” the per-kind arm is what lets us require a real label with no `""` sentinel on the tool surface. (The WIDE processing types `wideFlatItemSchema` / `wideEditUpdatesSchema`, used only to infer the flat/edit shapes the pipeline operates on, DO keep a required-with-sentinel `label` that `stripEmpty` collapses; that sentinel lives on the wide type alone, never on an arm.) Feature configs still group under nested objects (`validate: { expr, msg? }`, `repeat: { mode, count?, ids_query? }`) so each reads as one slot.
- `toolSchemas.ts` ā€” materializes the generator output once and exposes the stable Zod nodes the SA + `scripts/test-schema.ts` reuse.
- `planningSchemas.ts` ā€” describe-rich input schemas for the data-model planning tool (`generateSchema`) plus the shared per-form Connect block and the `case_type_record` shape `createModule` reuses. Separate from `toolSchemas.ts` because these describe whole-app structure, not per-field edits. Every shape is built so a wrong input can't parse: optional means omit (no `""`/`[]` sentinels on this surface), present strings are `min(1)`, and cross-field contradictions (a `relationship` with no `parent_type`, a connect block with no sub-config) are rejected with a message naming the fix. The planning tool is PURE: it records its plan in the conversation (the tool input is the plan; the result echoes a compact structured index) and writes nothing ā€” the app's design lives in the SA's chat text, and execution runs through `updateApp` + one `createModule` per module.
- `generationContext.ts` ā€” shared wrapper around the AI Gateway provider (the one model resolver for the SA, the document summarizer, and structured sub-gens), SSE stream writer, `LogWriter` (event log), and `UsageAccumulator` (cost ā€” the token-math `costEstimate` plus the gateway-metered `actualCost`, read per call from `providerMetadata.gateway.cost` via `gatewayActualCost`). Implements the shared `ToolExecutionContext` contract (`recordMutations`, `recordConversation`) on top of chat-specific internals (`buildEnvelopes` + the awaited-inline `commitBatch` ā†’ `commitGuardedBatch`, `emitConversation`, `emitError`) plus the shared `handleAgentStep(step, label)` helper used by the SA's `onStepEnd` callback. The only sanctioned way to commit a doc-mutating batch ā€” awaited inline through the guarded writer, `data-mutations` emitted only after the commit resolves (carrying its `seq`) ā€” and to write an agent-side log event.
- `errorClassifier.ts` ā€” the shared thrown-error taxonomy (network / provider / internal) both surfaces classify failures through.
- `blueprintHelpers.ts` ā€” pure `Mutation[]` builders the SA calls from its tool handlers (`addFieldMutations`, `addModuleMutations`, `renameFieldMutations`, etc.), plus the positional index ā†’ uuid resolvers every tool addresses entities through: `resolveModuleUuid` / `resolveFormUuid` / `resolveFormContext` / `resolveFieldByIndex`. **The SA's `moduleIndex` / `formIndex` are DISPLAY-order positions** (`sort-by-(order, uuid)` ā€” the same sequence `summarizeBlueprint` / `get_app` / `searchBlueprint` render), so these resolve through `ordered{Module,Form,Field}Uuids`, NOT `moduleOrder` / `formOrder` array position (which a same-parent reorder leaves untouched, so a raw index would address the wrong entity). A tool body that reaches into `doc.moduleOrder[moduleIndex]` is a defect; the `lib/agent/tools` grep guard in `lib/doc/__tests__/orderSequenceSweep.test.ts` fails the build on one. New-entity index reporting (`createModule` / `createForm`) resolves the born entity's SORTED position via `ordered*Uuids(...).indexOf(uuid)`, never `array.length - 1`.
- `contentProcessing.ts` ā€” sentinel stripping (`stripEmpty`) + case-type default merging (`applyDefaults`) + flat-to-discriminated reshape (`flatFieldToField`) for the `addFields` input. Domain-vocab throughout; case-type metadata (which uses CommCare-flavored `validation`/`validation_msg` on its property shape) is translated onto the field's nested `validate: { expr, msg? }` object at this one boundary. The reshape also flattens the SA's nested `repeat: { mode, count?, ids_query? }` into the schema's discriminated-union form (`repeat_mode` + variant-specific `repeat_count` or `data_source`). `unescapeXPath` is exported because `editPatchToFieldPatch` in `tools/editField.ts` needs the same XPath-entity normalization the add path applies.
- `documentExtraction.ts` ā€” the requirements-extraction CORE. `extractDocument` takes one document's bytes + kind and returns an `ExtractResult` (`{ extract, title, summary, truncated }`) from the official summarizer (GPT-5.6 Luna at `reasoningEffort: xhigh` ā€” `CONDENSER_MODEL` + `CONDENSER_PROVIDER_OPTIONS`). ONE structured call fills `extractDocumentSchema` ā€” `title` + `summary` FIRST, then the large `extract` LAST (schema field order is load-bearing: writing the extract last stops the model bleeding the trailing fields into the extract string mid-generation). Input shaping: text/markdown/csv decode to text and docx/xlsx convert to markdown (mammoth / SheetJS) and ride as the `prompt`; PDFs ride as a native document `file` block. The `extract` is FREE-FORM markdown grouped under `##` SECTIONS per `EXTRACT_SYSTEM` ā€” every requirement reproduced verbatim, unknown high-invention properties written `not stated`, every inference tagged `[derived]`; contradictions/omissions/unknowns surface under Conflicts / Gaps / Open questions and are never resolved. `title`/`summary` are specified ONLY by their schema `.describe()`s (the system prompt governs the `extract` alone). A structured call has no partial: truncation past `EXTRACT_MAX_OUTPUT_TOKENS` (128k) or a malformed response yields no parseable object, so extraction FAILS rather than returning half an extract. `title`/`summary` persist on the asset doc and ride the wire (`WireMediaAsset.extract`) for the file-manager label + preview header; the SA reading path uses only `extract`. The `condenser` is an `AttachmentCondenser` ā€” `createExtractionCondenser` (a standalone gateway-bound backend) for the upload-time extract route, or the chat run's `GenerationContext` for the resolve-step backstop (usage-tracked). `EXTRACTOR_VERSION` (in `@/lib/domain/multimedia`, beside the extract key it versions ā€” kept out of this mammoth-importing module so the pure key helper imports without dragging the office parsers into a caller's graph) keys the stored extract; bump it on any prompt/model/conversion change here to invalidate stale extracts with no migration. It is pure of HTTP + storage: the single-flight + persistence live in `documentExtractionStore.ts`, which both callers go through so a document is extracted ONCE, not per chat turn. `extractDocument` applies `normalizeExtractText` (below) to its result so a fresh extract is stored clean.
- `extractNormalization.ts` ā€” the over-escape repair, a dep-free leaf (no mammoth/xlsx, no I/O). `normalizeExtractText` decodes ONE JSON-string escape level, but ONLY on the unambiguous corruption signature (no real newline + a literal `\n`): a summarizer emitting a LARGE extract as a JSON string value under structured generation can double-escape its whole output (every newline ā†’ literal `\n`, every quote ā†’ literal `\"`), collapsing the extract to one physical line ā€” verified by isolation (the same input is clean as plain text, or as a small structured output; it's the OUTPUT size that degrades the escaping, not the input). It's applied at BOTH ends ā€” extraction (fresh extracts store clean) and read (`readyResult` in the store, and the preview GET route, so an extract stored before the repair existed is fixed on the way out, no re-extraction). Idempotent on a clean extract. The dep-free split lets the store/route ā€” and their tests ā€” import the real repair without dragging the office parsers in.
- `documentExtractionStore.ts` ā€” the durable, single-flight STORE for a document's extract, and the ONE entry point both the eager route and the chat backstop go through (so the lock lives in one place ā€” there is no second path that can bypass it). `ensureStoredExtract({ asset, documentKind, condenser, onInflight })`: a GCS-first fast path (a doc already extracted resolves in one read); on a miss, a fresh status read drives the pure `decideExtractAction` (reuse / wait-on-the-in-flight-job / claim+extract). `onInflight: "report"` (the eager route) returns `{ status: "extracting" }` so the badge poller gets a fast 202; `onInflight: "wait"` (the chat send-path) polls the in-flight job to completion and reuses its result ā€” never a second model call, and never slower than extracting fresh (the eager job started when the document was attached, before the send). `EXTRACTING_STALE_MS` bounds the wait and re-claims a job whose process died. `extractDocument` stays the pure bytesā†’text core it composes over.
- `resolveAttachments.ts` ā€” `resolveAttachments` turns a turn's attachment REFERENCES into model-ready content BEFORE the SA runs (the chat route calls it after `ctx` is built, before `createSolutionsArchitect`; `countDocumentsNeedingRead` gates the "reading documents" status ā€” only a document not yet extracted counts, so an already-read one resolves silently and never flashes it). The composer sends asset-id refs in message metadata; this walks EVERY message (not just the last ā€” that's the multi-turn-crash fix, history carries refs + resolved text, never raw `text/markdown` file parts the provider rejects), batch-loads the owner's assets once (a total load failure degrades to placeholders, never throws), and appends per ref: a document ā†’ its stored extract as a text part (via `ensureStoredExtract` with `onInflight: "wait"` ā€” it reuses an in-flight eager extraction, or extracts inline via `ctx` + persists when none is running), an image ā†’ its bytes as a data-URL file part for the SA's vision pass. Resolved parts are deduped by assetId and byte-identical across turns, so re-resolving history keeps the prompt cache hit. Every failure path falls back to a human-readable placeholder ā€” an attachment is never dropped. Cost note: `UsageAccumulator` pools every token (SA + any backstop sub-gen) at the run's `seed.model` (the SA model's) rate, so a backstop extraction that actually runs is costed at SA rates ā€” a pre-existing approximation. The store only meters when it runs the model itself; reusing an in-flight eager extraction (the common case for a document attached before the send) meters nothing, and eager upload-time extraction runs off the chat run.

## The write surface (server side)

The SA computes `Mutation[]` internally (via the helpers in `blueprintHelpers.ts`) and commits them through `tools/common.ts::guardedMutate` ā€” the one write path that runs the validity gate (`lib/doc/commitVerdicts.ts::mutationCommitVerdict` over `evaluateCommit`) BEFORE persisting via `ctx.recordMutations(mutations, doc, stage?)`. A batch that would introduce a validator finding fails the tool call with the findings in the `{ error }` envelope and persists nothing ā€” the agent self-corrects within its loop; an invalid intermediate never reaches Postgres or the stream, on chat and MCP alike. Multi-stage tools (`editField`'s convert ā†’ rename ā†’ patch) build their stages against local candidates and commit through `guardedMutateStages`, which gates the WHOLE sequence as one candidate and persists it as ONE save (`ctx.recordMutationStages` ā€” on MCP, a single transactional commit whose fresh-doc re-verdict evaluates the concatenated batch) ā€” a rejection, optimistic or transactional, leaves zero committed prefix, so "a rejected call saved nothing" holds with no multi-stage asterisk. A transactional rejection that escapes a tool body (`BlueprintCommitRejectedError`) surfaces as the standard validity `invalid_input` envelope via `toMcpErrorResult`. There is no post-hoc fix loop and no finishing step: the gate at every commit plus atomic creation is the whole validity story.

Clients of the stream (the interactive builder) receive `data-mutations` events and feed the payload straight into `docStore.applyMany(mutations)` ā€” no translation, no reconstruction. The agent and the user speak the same mutation API.

`data-done` still carries the full `PersistableDoc` ā€” emitted by the chat route's drain-end finalize, the one full-doc reconciliation point that guarantees the client doc matches the run's final persisted snapshot, whatever streaming raced it. Nothing else emits full docs on the live path.

## SA prompt caching

OpenAI prompt caching is implicit ā€” no cache option in the provider options. The provider manages the breakpoint (placed near the latest message) and reuses any cached prefix, so the system prompt + stable history stay cached across requests within a session. Keep the prefix stable: churn in the system prompt or early messages silently kills the hit rate.

Cache TTL is 30 minutes ā€” the guaranteed retention floor (`CACHE_TTL_MS`); OpenAI may keep a prefix longer. The route uses a client-reported timestamp to choose the message strategy: within the window, full history is sent; after expiry, only the last user message goes (one-shot edit). Edit-vs-build mode is an ORTHOGONAL decision: app-exists picks the prompt + tool set AND the model (edit mode excludes the planning tools and runs `SA_EDIT_MODEL`; a build runs `SA_BUILD_MODEL` ā€” see "Two tool groups" below), while the cache window picks the message strategy. The two never collapse into one flag.

## Provider options shape

SA shape: `{ reasoningEffort, reasoningSummary: 'auto' }` under the `openai` key, plus `GATEWAY_PROVIDER_OPTIONS` (`disallowPromptTraining: true` ā€” user content never trains a provider model ā€” and `caching: "auto"` for the gateway's automatic prompt caching) under the `gateway` key on EVERY call (SA, sub-gens, extraction, scripts). `reasoningSummary: 'auto'` is required for human-readable reasoning summaries to stream back as `reasoning-delta` parts. The AI SDK's Zod schema silently strips misplaced/unknown fields, so a typo'd option appears to work and silently doesn't reach the wire ā€” always check provider options against `OpenAIResponsesProviderOptions` via `satisfies` on the literal, not an annotation; never type them `Record<string, JSONValue>`.

## Two tool groups: planning + shared

Tools split into a planning set (build mode only: `generateSchema` ā€” pure, zero doc writes; the app's design itself lives in the SA's chat text) and a shared set (all modes: `askQuestions`, `searchBlueprint`, `getModule`, `getForm`, `getField`, `addFields`, `editField`, `removeField`, `updateApp`, `updateModule`, `updateForm`, `createForm`, `removeForm`, `createModule`, `removeModule`, `addCaseListColumns`, `updateCaseListColumn`, `removeCaseListColumn`, `reorderCaseListColumns`, `addSearchInputs`, `updateSearchInput`, `removeSearchInput`, `reorderSearchInputs`, `setCaseListFilter`, `setCaseSearchAdvanced`, `setCaseSearchDisplay`, `attachFieldMedia`, `attachOptionMedia`, `setMenuMedia`, `setAppLogo`, `listMediaAssets`, `removeMediaAsset`). When the app already exists, the planning tool is excluded.

**No singular add-tool has a plural twin.** Anything the SA adds one-or-more of in a list ā€” fields, case-list columns, search inputs ā€” has exactly one list-taking tool (`addFields` / `addCaseListColumns` / `addSearchInputs`); one item is a length-1 array. Don't reintroduce a singular `addField` / `addCaseListColumn` / `addSearchInput`: a redundant tool only burns context and invites the SA to add items one call at a time. The chat route's history strip drops any tool-use part naming a tool absent from the current set (`Object.keys(sa.tools)`), so removing or renaming a tool can't orphan an old thread's references on a live-cache edit. Mutation tools return a human-readable success `message` (alongside a UI-only `summary` and any addressing `uuid` ā€” see the return-contract section), so the SA trusts its own edits from the prose without re-reading the blueprint.

**Every tool-surface XPath string crosses the text ā†’ AST parse boundary before it lands on the doc.** Field expression slots parse in `tools/shared/fieldAssembly.ts`'s second pass (batch-aware: a reference to a field landing in the same call resolves to an identity leaf) and in `editField`'s `editPatchToFieldPatch` (`parseXPathForField`); the Connect-block XPath slots (`assessment.user_score`, `deliver_unit.entity_id`/`entity_name`) parse in `tools/shared/connectInput.ts::buildConnectConfig` ā€” `updateForm` calls it against the owning form, the creation tools against their assembly's batch resolver. Never bridge a tool input into a `ConnectConfig` (or any AST-typed slot) with a cast: the commit gate reads expressions through the total-reader projection and passes a raw string clean, but the NEXT load's Zod gate (`appDocSchema` ā†’ `xpathExpressionSchema`) rejects it and the app stops loading. The construction fuzz's per-op `blueprintDocSchema` parse and `atomicCreation.test.ts`'s persisting-mock are the standing tripwires for this class.

**Structural creation is atomic.** `createForm` takes the form's `fields` (required ā€” the same per-kind items `addFields` takes, assembled through the shared `tools/shared/fieldAssembly.ts` pipeline) and `createModule` takes `forms` (each with `fields`) plus `case_list_columns` plus `case_type_record` (the case-type record from the data-model plan, required when the module's type is new to the app ā€” a record can only land WITH the module that satisfies its validator obligations, which is what makes `MISSING_CHILD_CASE_MODULE` unreachable from the tool surface; the assembly sees the record in the same call, so catalog defaults seed this module's own fields), so an entity lands together with what makes it sound and complete in ONE gated batch. This is what keeps creation alive under the single rule: a lone `addForm` would introduce `EMPTY_FORM` and a lone case-typed `addModule` would introduce `NO_FORMS_OR_CASE_LIST` + `MISSING_CASE_LIST_COLUMNS` ā€” with the atomic shapes, every rejection's findings are satisfiable by adjusting the same call. Case list REFINEMENT (sort, filter, search inputs, later column edits) stays the responsibility of the case-list-config tools once the module exists; `createModule`'s `case_list_columns` exists so the module can be born complete, not as a second authoring surface. `updateModule` carries `name` + `case_type` + an optional `case_list_columns` seed (applied only when the module has none ā€” the case-type flip on a module with forms obliges columns, so the same call can satisfy them). `updateApp` is the app-level twin: `name` + `connect_type` (enabling Connect is rejected while NO form carries a connect block ā€” the app-level participation floor ā€” so a new Connect build flips the type FIRST, on the empty app; forms then opt in per form via their `connect` blocks). Case-search authoring is the parallel responsibility of the case-search-config tools (`setCaseSearchAdvanced` / `setCaseSearchDisplay`). This keeps the typed `Column` / `SearchInputDef` / `CaseSearchConfig` shapes end-to-end on every authoring path. Media authoring is the parallel responsibility of the media tools in `tools/media/` ā€” the generic mutation tools (`addFields` / `editField` / case-list-config) carry NO media slot, so the SA can't mint or reference an asset id there. See "Media authoring" below.

### Case-list authoring ā€” atomic ops + uuid handles

The `caseListConfig` shape has three slots ā€” `columns`, `filter?`, `searchInputs` ā€” and the SA tool surface reflects that:

- **Two arrays decompose into ops.** Each of `columns` and `searchInputs` has a list-`add` (`addCaseListColumns` / `addSearchInputs`) plus `update` / `remove` / `reorder`. The `add` takes a list ā€” minting a uuid per item and surfacing them positionally in `result.uuids` ā€” so a module's whole set authors in one call; `update` / `remove` consume a single uuid as the addressing key. Reference the minted uuids directly on follow-up edits without re-reading.
- **One wholesale tool for `filter`.** A filter is one Predicate, so a wholesale `setCaseListFilter` (with a `null`-clears convention) fits.
- **Read tools surface uuids.** `getModule`'s `case_list_config` returns the structured config verbatim ā€” every column and search input carries its `uuid`. `searchBlueprint`'s `case_list_column` and `search_input` matches surface the entry's `uuid` plus the owning module's `containerUuid`. `summarizeBlueprint`'s prompt-time module summary lists each case-list entry's uuid alongside its kind + label so the SA inherits the handles after a fresh-session edit-mode resume.

The `update*` tools accept the full body shape (kind + per-kind required fields + common optional slots) ā€” partial-patch shapes don't fit the 8-optional ceiling on the column union (the interval arm alone has six per-kind fields), and switching between `simple` / `advanced` search-input arms requires a different field set anyway.

The case-list-config tools accept the typed AST shape directly via Zod ā€” `Column`, `Predicate`, `SearchInputDef` ā€” pulled from `lib/domain/predicate` and `lib/domain/modules`. The ops route their mutation emission through `addColumnsMutation` / `updateColumnMutation` / `removeColumnMutation` / `reorderColumnsMutation` (and the search-input parallels) in `lib/agent/blueprintHelpers.ts`, which emit the GRANULAR per-item kinds (`addColumn` / `updateColumn` / `removeColumn` / `moveColumn`, keyed by item uuid + a fractional `order`) ā€” NOT a wholesale `updateModule{caseListConfig}` ā€” so two members editing different columns/inputs merge by construction (`reorderColumnsMutation` assigns fresh ascending `order` keys to the requested sequence). The builders still validate through the pure generic primitives (`replaceByUuid` / `removeByUuid` / `reorderByUuid`) that live alongside the SA-boundary input schemas (`columnInputSchema`, `searchInputDefInputSchema`, both with `uuid` AND `order` omitted ā€” the tool mints the uuid and the gesture/diff computes the `order`) at `tools/case-list-config/shared.ts`. `setCaseListFilter`'s filter rides the granular `setCaseListMeta` kind for the same reason. `blueprintHelpers.ts` owns the agent-specific `Mutation[]` builders.

### Case-search authoring ā€” wholesale per cluster

The `caseSearchConfig` shape is a settings bag, not an addressable list, so its tool surface is two wholesale-replace tools rather than atomic ops:

- **`setCaseSearchDisplay`** owns the display cluster: `searchScreenTitle`, `searchScreenSubtitle`, `searchButtonLabel`, `searchButtonDisplayCondition`.
- **`setCaseSearchAdvanced`** owns the advanced cluster ā€” niche search-side filters that most authors never reach for. The cluster carries the `excludedOwnerIds` slot (value expression naming owner ids whose cases are excluded from search results). The abstract "advanced" framing scopes the tool to its role (niche filters), not its contents. The slot translates at suite-XML emission time to CCHQ's wire field `commcare_blacklisted_owner_ids` ā€” Nova's authoring vocabulary uses `excludedOwnerIds`, the wire token is CCHQ-controlled.

Each tool replaces its own cluster in one call and preserves the other cluster byte-identically ā€” pick the OTHER cluster's slots forward off the existing config, layer the input over them. Wholesale-with-`null`-clears semantic mirrors `setCaseListFilter`: every cluster slot is required-and-nullable on the SA boundary; `null` clears, non-null sets. That keeps the per-tool optional count at zero (well under the tool-schema 8-optional ceiling) and removes the "absent vs null" ambiguity the SA would otherwise have to resolve.

Cross-binding rule ā€” search inputs themselves stay on `caseListConfig.searchInputs` (one source of truth across both screens). Author them through the case-list-config search-input family (`addSearchInputs` / `updateSearchInput` / `removeSearchInput` / `reorderSearchInputs`), never inside the case-search-config tools. The case-search-config tools intentionally do not carry a `searchInputs` slot.

The case-search-config tools accept the typed AST shape directly via Zod ā€” `Predicate`, `ValueExpression` ā€” pulled from `lib/domain/predicate`. The wholesale tools emit a single `updateModule` mutation patching `caseSearchConfig` via `updateModuleMutations`. The shared `moduleNotFoundResult` helper (used by every module-addressing SA tool family) lives at `tools/shared/moduleNotFoundResult.ts` so both families consume one Elm-style error shape; `tools/case-list-config/shared.ts` re-exports it for the existing case-list-config call sites.

### Media authoring ā€” dedicated carriers + the asset library

The generic mutation tools deliberately omit every media slot (`toolSchemaGenerator.ts`'s `saOptionSchema` drops `media`; the field-edit schema carries no `*_media` key). The SA can neither mint nor discover an asset id from those surfaces, so a media slot there would only let it write a dangling reference. Media authoring lives in its own `tools/media/` package:

- **Four doc-mutation tools** attach asset ids to carriers ā€” `attachFieldMedia` (a field's `label`/`hint`/`help`/`validate_msg` message-media bundle), `attachOptionMedia` (a select option's `media` bundle), `setMenuMedia` (menu tiles ā€” module AND form, mixed in one call: `icon` + `audioLabel`, image + audio, no video), and `setAppLogo` (the blueprint-root `logo`). Each persists via `ctx.recordMutations` and returns a `MutatingToolResult` summary. Every tool whose carrier repeats is **batch-shaped** (a list of items/attachments, one entry is a length-1 array ā€” the no-singular-twin rule applied to media), and a batch is **all-or-nothing**: every item must resolve and every set slot must pass the asset verdict before the single gated commit, else `{ error }` names every offending item and nothing is written. `attachFieldMedia` gates slot-vs-kind through `fieldKindDeclaresKey` (the schema key set, NOT `key in field` ā€” an unset optional slot is absent as an own property even on a kind that supports it). `setAppLogo` is the ONLY writer for `doc.logo`.

  **The menu-tile `icon` slot also takes a built-in icon slug** ā€” each `setMenuMedia` item's `icon` is a `nullableIconSlot` (`tools/media/shared.ts`): a union of the curated slug enum (the `target: module` arm offers `MODULE_ICON_SLUGS`, the `target: form` arm `FORM_ICON_SLUGS`) and any string, so the SA can pass a built-in slug (e.g. `household`) with no `list_media_assets` round-trip, OR an uploaded image's asset id, OR `null` to clear. `resolveIconInput` disambiguates by catalog membership (slugs vs `randomUUID()` asset ids can't collide): a slug ā†’ the reserved `nova-icon:<slug>` ref with NO attach expectation (built-ins have no library row ā€” they resolve from the shipped set at emit, so the asset-row verdict must skip them, which an empty expectation list does); any other string ā†’ an uploaded id with the standard image expectation. The shared `lib/media/builtinIconAssets.ts` seam makes a built-in ref emit exactly like an upload (see `lib/media/CLAUDE.md` Ā§ built-in library icons). There is deliberately ONE place to set a module/form icon: do NOT add an `icon` slot to `createModule`/`createForm`/`updateModule`/`updateForm` (that's the no-media-on-the-generic-tools rule). The prompt's `## Media` block points the SA at `setMenuMedia` ā€” and carries the vary-icons-within-a-screen guidance, which leans on the whole menu landing in one call.

  **Media slots use dedicated, clear-safe mutation kinds ā€” never the generic `update*` patch reducers.** `attachFieldMedia` ā†’ `setFieldMedia` (one per attachment), `setMenuMedia` ā†’ `setModuleMedia` / `setFormMedia` (one per tile, by its arm), `setAppLogo` ā†’ `setAppLogo`. The reason is a wire hazard: the SA streams mutations to the client as JSON, and `JSON.stringify` DROPS keys whose value is `undefined`. A clear encoded as an `updateField`/`updateModule`/`updateForm` patch of `{ <key>: undefined }` arrives at the client's `applyMany` as `{}` ā€” a no-op ā€” so the stale asset ref survives and the client auto-saves it back over the SA's correct clear. The dedicated kinds carry an explicit on-wire `null` (which survives JSON) and map `null ā†’ undefined` INSIDE the reducer, so both set and clear cross the wire intact. This is NOT folded into the generic reducers as a "null-means-clear" rule: `setConnectType`'s slot is genuinely `.nullable()` and stores `null` as a real value, so a generic rule would corrupt it ā€” the clear-safe behavior stays scoped to these media-only kinds. `attachOptionMedia` follows the same granular discipline for a different reason: it emits a per-item `updateOption { fieldUuid, uuid, option }` (the rebuilt option carries or omits the `media` key), so a concurrent edit to a DIFFERENT option of the same field merges instead of clobbering ā€” a wholesale `updateField{options}` patch would overwrite the whole array. A clear rides that same `updateOption` (the rebuilt option simply omits `media`), so it needs no `null` sentinel.
- **Two library tools** (`ReadToolResult`-shaped ā€” no doc mutation): `listMediaAssets` (load-bearing ā€” how the SA discovers the asset ids the attach tools need; reuses `listReadyAssetsForProject` + `toWireMediaAsset`) and `removeMediaAsset` (deletes the Postgres row + the GCS bytes + the document-extract sibling, but refuses with an Elm-shape error naming the carrier(s) when the asset is still referenced ā€” in the in-hand working doc OR any live app in the asset's Project ā€” the reference guard). The deletion mechanics live in `lib/media/assetDeletion.ts` (`findAppReferencesToAsset` + `purgeAssetStorage` + `describeCarrier`), shared with the browser `DELETE /api/media/[assetId]` route so the SA tool and the file-manager delete can't drift; the tool adds only the in-hand-working-doc check the route can't see. `findAppReferencesToAsset` does NOT scan the whole Project up front ā€” it reads the asset's `referencingAppIds` reverse index (the apps whose persisted blueprint references it, maintained append-only by the blueprint writers' `syncMediaReferences`) and re-walks ONLY those 0ā€“2 candidates to confirm a live reference + name the carrier. Append-only means a candidate can be stale, so the re-walk is what proves it (a stale one drops out); an un-backfilled row (`referencingAppIds` absent) falls back to the full Project-wide scan (`listApps(projectId)`). This turned the guard from a measured ~8s on an 83-app account into ~0ms for an unreferenced asset.

**Every attach runs the at-source asset verdict before its gated commit.** The four doc-mutation tools route through `tools/media/shared.ts::attachGuardedMutate` ā†’ `lib/media/attachVerdicts.ts::mediaAttachVerdict`: the asset must exist in the app's Project, be `ready`, match the slot's kind, and keep the app's referenced-ready-media aggregate inside the export ceiling ā€” a failed expectation returns the `{ error }` envelope with nothing written, so a committed reference can't dangle (the verdict module's header walks why attach-time checking holds: referenced-asset deletion is guarded, `ready` is terminal, `project_id` is the access gate, kind is immutable). Clears carry no expectations and skip the asset read. On chat the verdict is a pre-commit read (the run owns its doc); on MCP the expectations ride `guardedMutate` ā†’ `recordMutations` into the transactional commit, which re-applies the per-asset judgment to rows read in the SAME transaction that re-verdicts the batch ā€” a delete that COMMITS before the attach's read fails the re-verdict and the attach refuses. The reverse interleaving survives on both surfaces (the delete's reference guard reads the after-commit-updated reverse index outside any transaction, so a guard read just before the attach commit lands approves a delete whose row drops just after): the ref it strands is what the export boundary's media rules (`lib/media/boundaryValidation.ts`) and the legacy repair pair's `--media` arm exist for, alongside their standing role covering legacy refs and ops disasters.

The MCP-only `upload_media_asset` (`lib/mcp/tools/uploadMediaAsset.ts`, hand-registered ā€” neither doc- nor app-scoped, so it can't ride the shared adapter) is the bytes-inline upload path for MCP clients that can't run the browser's hash ā†’ signed-PUT ā†’ confirm flow; it runs the same `validateMediaBytes` pipeline and stores via `uploadAssetBytes`. The shared media tools register in BOTH `lib/mcp/server.ts`'s `SHARED_TOOLS` and `lib/agent/solutionsArchitect.ts`'s `sharedTools` (the chat surface keeps its own parallel list).

## Shared-tool return contract

Every `lib/agent/tools/<name>.ts` `execute` returns one of two tagged shapes (`tools/common.ts`):

- `MutatingToolResult<R>` ā€” `{ kind: "mutate", mutations, newDoc, result }`. Tool body has already persisted via `ctx.recordMutations` before returning. On success `result` is `{ message, summary, ...extras }`: `message` is the prose the LLM reads; `summary` (`ToolCallSummary` in `tools/shared/toolCallSummary.ts`) is UI-only presentation for the chat transcript ā€” the names the tool already resolved, exposed as discrete fields so the transcript renders a friendly action + location breadcrumb rather than the prose. The MCP projector strips `summary` from the wire (and unwraps a now-`{ message }`-only object back to the bare string), so the MCP shape is unchanged. `extras` is the per-tool addressing payload (e.g. a minted `uuid`). On failure `result` is `{ error }`. The no-change branches of `removeModule` / `removeForm` (target didn't exist) return a bare `string` ā€” there's no entity to summarize, and the transcript falls back to that explanatory prose.
- `ReadToolResult<R>` ā€” `{ kind: "read", data }`. Pure read, no persistence. The planning tool (`generateSchema`) is read-shaped: its plan lives in the conversation, not on the doc.

The `kind` discriminator is the contract two consumers dispatch on:
- Chat-side `wrapMutating` / `wrapRead` in `solutionsArchitect.ts` destructure the shape and surface only the inner payload to the AI SDK tool.
- MCP `projectResult` in `lib/mcp/adapters/sharedToolAdapter.ts` switches on `kind` to project to the wire envelope.

Adding a new shared tool: pick a shape, return it tagged. A future third shape requires updating both consumers ā€” the exhaustive `switch` in `projectResult` is the compile-time tripwire.
