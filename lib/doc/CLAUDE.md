# lib/doc — Builder document store

The normalized, undoable source of truth for the blueprint domain. Every module, form, and field the builder edits lives here, keyed by UUID.

## Boundary rule

The store is private. Consumers must go through the named domain hooks in `hooks/` — never import the store file directly from outside this package.

Consumers get narrow, memoized hooks with predictable selector shapes; no component passes a raw selector function to a Zustand hook for this store. This boundary keeps undo/redo semantics sane and lets the internal store shape evolve without touching call sites.

## The write surface

`applyMany(mutations: Mutation[]): MutationResult[]` is the mutation-applying write action on the store. `renameField` and `moveField` produce a `MutationResult` entry at their array position — `FieldRenameMeta` (rewrite counts: PROSE hashtag rewrites plus the case-property cascade's structural leaf renames; expression slots store identity leaves, so form-local renames/moves rewrite zero of them) and `MoveFieldResult` (cross-level sibling-id dedup info) respectively. Every other mutation produces `undefined`. There is no `apply` / `applyWithResult` — single-mutation dispatches wrap as `applyMany([m])` with `[result]` destructuring for metadata.

`commitDoc(next)` is its gated-dispatch twin: `useBlueprintMutations`' gate runs the batch through `prepareMutationCandidate` once, then `evaluatePreparedMutationCandidate` validates without receiving or re-running the mutations; on a pass it commits that exact candidate directly. One reducer run per UI dispatch and authoritative transaction attempt, and the committed doc is exactly the doc the gate validated (load-bearing for `duplicateField`, whose reducer mints a fresh clone uuid per run). `duplicateField` is UI-only: the local doc diff persists explicit `addField` payloads carrying the clone identities, and `assertPersistenceSafeMutationIdentities` rejects the reducer-minting convenience at an authoritative boundary. One `set()` either way, so zundo records one undo entry. Every other writer (agent stream, replay, compound session actions) stays on `applyMany`.

The public `Mutation` union is fine-grained. There is no `replaceForm`: wholesale form replacements decompose into `updateForm` + `removeField × N` + `addField × M` at the emission boundary — live generation produces that sequence directly via `lib/agent/blueprintHelpers.ts`. There are no `notify*` mutations: prose hashtag rewrites + sibling-id dedup are reducer side-effects of `renameField` / `moveField`, surfaced to UI as `MutationResult` return values. The `notifyMoveRename` helper in `mutations/notify.ts` is a UI toast emitter consumers call with the `MoveFieldResult` they destructure from `applyMany`'s return array — not a mutation kind. Rolling deploys make discriminators a compatibility boundary: a post-deploy event must use a kind an open pre-deploy tab and an old server already recognize. `mutationSchema` is therefore the carrier-blind external/rolling envelope, while `canonicalMutationSchema` is the full-vocabulary durable replay/log envelope and the source of the public `Mutation` type; lookup-backed select intent may appear only as top-level `addField.optionsSource` / `updateField.optionsSource`, never inside a legacy fallback or on another discriminator. New case-workspace semantics therefore extend existing kinds with optional top-level fields plus origin-compatible fallbacks. `updateColumn.visibilityPatch` changes one surface slot and `preserveVisibility` protects fresh visibility during content edits; every fallback column omits the new nested `listOrder` / `detailOrder` keys because origin/main's nested column schema is strict. `moveColumn.surfaceOrderPatch` writes only Results or Details in a new reducer while its generic `order` is the old-reducer fallback; `addColumn.surfaceOrders` reconstructs both new keys after an old-shape nested column is added. `updateModule.ensureCaseListConfig` idempotently births the empty case-list container without replacing peer contents, while its patch carries the empty snapshot an old reducer applies. Search presence, final-input cleanup, and owner-only intent use the same `updateModule` extension pattern. Dormant case-operation add/update/remove/move uses `updateForm.caseOperationChange`: the current reducer applies the semantic edit, while the ordinary strict form patch is the old receiver's harmless fallback and carries no operation state. Diffing emits the same extension (including an explicit update when an operation's optional `order` is cleared), and stale update/remove/move events are no-ops rather than birthing an empty operation array. `addModule` is built only through `addModuleMutation.ts`, which strips current-only nested column/Search slots into top-level hydration values before retaining a strict origin-compatible nested module. The frozen two-direction matrix in `__tests__/mutationRollingCompatibility.test.ts` proves new payload → old parser/reducer fallback, raw new SSE event → old reducer dispatch, and new reducer semantic merge.

**Clearing an optional slot uses an explicit `null`, never `undefined`.** A clear must survive two serialization hops: the SSE `data-mutations` wire and the persisted jsonb (both are `JSON.stringify`, which DROPS `undefined`-valued keys). An `undefined`-valued clear applies in memory but round-trips through neither — it never reaches the client doc store, and the next auto-save re-writes the stale value. `null` survives both. So `updateField`, `updateModule`, and `updateForm` all blank a property by setting it to `null`: each reducer applies its patch key-by-key, deleting the slot on `null` or `undefined` (only `null` round-trips). Their patch schemas make the CLEARABLE (`.optional()`) slots `null`-accepting (`clearablePartialPatch` in `types.ts` mirrors `partialOf` in `lib/domain/fields`), so a `null`-clear parses; the edit-tool schema reserves `null` for "clear". This null-as-delete rule is scoped to these three patch reducers — do NOT extend it to `setConnectType`, whose slot is genuinely `.nullable()` and stores `null` as a real value (verify any new clearable slot the same way: a slot is null-as-delete-safe only when every reader treats `null` identically to absent — `form.connect` qualifies, all its readers guard `if (!connect)`). A required slot (`id` / `name` / `type`) stays non-nullable in the patch schema, so a stray `null` for it is a parse error rather than a corrupting assign — `updateModule` / `updateForm` apply their patch without a final whole-entity re-parse. Media slots — field message media, module/form menu media, app logo — clear through dedicated mutation kinds (`setFieldMedia` / `setModuleMedia` / `setFormMedia` / `setAppLogo`) instead, each carrying `media: Media | null` mapped to `undefined` INSIDE the reducer, because media is deliberately excluded from the generic field-edit surface. (A concrete array survives JSON whole, so a wholesale `{ options: [...] }` / `{ caseListConfig: {...} }` patch can clear a NESTED key by omitting it from the rebuilt object — that wholesale-replace path needs no per-key null and stays on `updateField` / `updateModule`.)

## Doc-shaped helpers colocated here

Pure functions that read or produce a `BlueprintDoc` (or a subtree of one) live in this package alongside the store:

- `fieldPath.ts` — path ↔ uuid resolution (`resolveFieldByPath`, `getFieldPath`). The doc stores fields by uuid; blueprint mutations and the agent tool surface operate in path strings.
- `scaffolds.ts` — atomic born-valid `Mutation[]` builders for the builder's in-tree "add module / add form" affordances (`caseListModuleMutations` / `surveyModuleMutations` / `formScaffoldMutations`). The UI twin of the SA's `createModule` / `createForm`: each lands the entity TOGETHER with the minimal contents the validator requires, so the whole batch passes the commit gate as one candidate. A case-list module is born as a VIEWER — a `caseListOnly` module with a `Name` column and its case type declared in the catalog, NO form (a name-only registration form is valid wire, so there's no reason to force one); a new case type is declared via `declareCaseTypeMutations` (the granular `declareCaseType` kind, idempotent), since a formless viewer has no `case_name` field to auto-register it. `scaffolds.ts` also owns the case-type DECLARATION CHOKEPOINT (`declareCaseTypeForField`): the reducer does not auto-create an absent type on a field write (which would clobber a concurrent declaration), so every `case_property_on`-setting surface — the SA add/edit assembly, the MCP field handlers, `useBlueprintMutations` add/edit, `caseListModuleMutations`, and `formScaffoldMutations` — prepends a `declareCaseType` for a field writing to an undeclared type, and the gate rejects a leftover writer via `CASE_PROPERTY_ON_UNKNOWN_TYPE`. A per-surface test (`__tests__/multiplayerMerge.test.ts`) exercises each field-birth chokepoint against an absent-type doc so a new surface that skips the declare fails CI. A survey module is born WITH one survey form (a single text question) — NOT bare: a module with no forms and no case list is a hard, build-blocking error in CommCare (`NO_FORMS_OR_CASE_LIST`, regardless of case type), so the only valid formless shape is a `caseListOnly` viewer. `formScaffoldMutations` lands a form with its default first field (`registration` → a `case_name` writer reading the module's case type; otherwise one text question). Dispatched through `useBlueprintMutations`' `createCaseListModule` / `createSurveyModule` / `createForm`.

`scaffolds.ts` also owns the CREATION TEMPLATES — what an app can be BORN holding, as opposed to what the builder adds to a live one. `blankAppMutations` + `BLANK_APP_NAME` are the blank app the `createBlankApp` Server Action (`app/(app)/build/actions.ts`) hands to `createApp`'s `seedMutations`; it's exactly the `surveyModuleMutations` shape (one survey module + one survey form) — the smallest EXPORT-ready app, and the minimum that satisfies `docHasData`, without which the builder bounces back to the centered chat. `BLANK_APP_NAME` is deliberately NOT `UNTITLED_APP_NAME` — that one is the display fallback `denormalize` writes into the summary row for a doc whose name is blank; this is a real persisted `appName`, and a blank app needs one because `EMPTY_APP_NAME` blocks export and there is no SA run to name it. `createApp` (`lib/db/apps.ts`) calls the template and `prepareMutationCandidate` exactly once outside its retryable transaction, then evaluates that prepared candidate with one locked lookup-definition context AND runs `exportReadinessFindings` before the app root, exact lookup edges, and entity rows commit atomically. The whole-doc boundary oracle matters because the introduced-error gate is delta-based and would let a pre-existing `NO_MODULES` stand; "a templated app is born export-ready" is enforced, not merely tested.
- `order/` — the order-key model: `keys.ts` (the fractional-key primitives — `keyBetween` REQUIRES an ordered interval `lo < hi` (null ≡ ±∞) and THROWS on a degenerate one, `keysBetween` mints a run inside one, and `keysForSlot` is the collision-safe slot layer EVERY insert-between gesture routes through: it widens past a run of equal-keyed siblings — a legit uuid-tie-broken rest state — to a distinct bound so the interval is never empty, and both the SA anchor and the builder drag land identically at a collision; `deriveKeyAtIndex` is the single-key wrapper), `backfill.ts` (deterministic position-seeded backfill of absent `order` keys + select-option uuids — same legacy doc → byte-identical keys on client and server; collision-tolerant, so an inverted/equal legacy bound appends rather than tripping the precondition), `compare.ts` (`bySortKey`, plus the total `byListColumnOrder` / `byDetailColumnOrder` surface comparators), `append.ts` (the doc-shaped `appendOrderKey` / `sequenceOrderKeys` / `sortedOrderKeys` over `bySortKey`), `options.ts` (`keyedOptions` — mints a born select field's option `uuid` + `order`, shared by the SA assembly, the builder add gesture, and the builder's convert-to-select gesture; a text→select `convertField` carries its born options ON THE MUTATION, minted at the batch-building layer, because the select schemas require `.min(2)` options the source kind can't hold and the reducer must stay deterministic for replay), `fieldSlot.ts` (`orderKeyForFieldSlot` — the one slot → field-order-key computation, shared by the builder's drag/insert dispatches and the SA's `moveField` tool so both surfaces mint the same key for the same gesture), `columnSurface.ts` (plans one Results / Details gesture as ONE surface mutation), and `searchInput.ts` (does the same for Search). **Sequence is DERIVED, not stored**: membership arrays (`moduleOrder` / `formOrder[m]` / `fieldOrder[p]`, and the `columns` / `searchInputs` / `options` arrays) are sets whose position is NOT authoritative. Generic sequences use `sort-by-(order, uuid)`; case-list columns independently use `listOrder ?? order` for Results and `detailOrder ?? order` for Details. Each gesture computes one absolute fractional key against the latest doc and the reducer stores it VERBATIM; a same-parent reorder sets only the relevant key and leaves membership arrays untouched, while content edits preserve current order and visibility slots so stale content cannot clobber a concurrent reorder or hide/show. Backfill runs at every hydration boundary (`store.load`, `loadAppBlueprint`) before the index builds. Order keys (and option uuids) never reach CommCare — consumers sort and drop them at the emission boundary.
- `fieldWalk.ts` exposes `orderedModuleUuids` / `orderedFormUuids` / `orderedFieldUuids` — the one place every SEQUENCE walk sorts through `bySortKey` (the SA positional resolvers, `summarizeBlueprint`, `searchBlueprint`, the wire emitters, the builder render hooks + search filter, the chat signal-grid). `__tests__/orderSequenceSweep.test.ts` is a two-part grep tripwire: the enumerated `SEQUENCE_CONSUMERS` list fails the build if a listed consumer loses its `bySortKey` / `ordered*Uuids` sort, AND a maintenance-free scan of the whole `lib/agent/tools` tree fails if any SA tool resolves a positional `moduleIndex` / `formIndex` by RAW `moduleOrder` / `formOrder` array position instead of the sorted `resolveModuleUuid` / `resolveFormUuid` helpers — the SA's positional vocabulary is DISPLAY order, so a raw index addresses the wrong entity after a same-parent reorder.
- `fieldParent.ts` / `fieldWalk.ts` / `searchBlueprint.ts` / `predicates.ts` — reducer-shaped utilities the store and UI share.
- `navigation.ts` — doc-aware `Location` resolution for the routing hooks.
- `kindConversionCascade.ts` — the property-centric conversion plan every kind-conversion surface consults (`planKindConversion`): a case-bound data-type flip — ANY flip, string-scalar, temporal, numeric, or the select↔text reshapes — converts every peer FIELD writer DERIVING the property's current type (derived-type selection, so a barcode/secret writer on a text property counts; one whose kind can't reach the target BLOCKS the plan with the peer named — the batch would otherwise bounce off the gate misreading a healthy property as broken) in the same gated batch, and re-declares a stale declared `data_type` via `setCaseProperty` (no field edit can otherwise satisfy `FIELD_KIND_PROPERTY_TYPE_MISMATCH`'s "change the property's data_type"). A case-operation writer blocks the gesture explicitly: its expression is a first-class writer but has no mechanical field-kind conversion, so the author must update/remove that operation before changing the property type. Stored rows are never the plan's business — `applySchemaChange` reshapes string↔array flips totally and casts every other transition per row, parking what cannot carry (holding its case). A flip whose per-row cast can fail (`castCanFail` in `lib/domain/casePropertyTypes.ts`) carries a `dataLossRisk` verdict on the plan; the CONSUMERS gate consent on it — the builder's `ConvertImpactDialog` and the SA `editField`'s needs-confirmation round both count the real impact via the store's `conversionImpact` preview before dispatching — while the committed mutations stay consent-free (replay purity: undo and historical replay re-run the migration unconditionally, and the review surface is the recovery path). Consent is scoped to the CONVERT gesture by design: a retype reached without one — a `case_property_on` retarget onto an undeclared property with rows of another type, a hidden writer's expression inference — has no gesture to hook, so it stays write-time-detected, parked + held, and reported (the toast, the parked note, Data to review), per the #252 design record. A conversion to `hidden` (agreement-exempt) converts only the addressed field and PINS an undeclared entry to the source kind's type when it was the last typed writer, so later expression inference can't retype the property under its existing rows. Peers resolve through the reference index's `declarersOf`, never an unindexed whole-doc search; callers whose same call retargets/clears `case_property_on` pass the field with that override applied. Consumed by the SA/MCP `editField` and `useBlueprintMutations.convertField` — same batch-building-layer-cascade discipline as `caseTypeRetirement`.
- `unwrittenProperties.ts` — the informational "where does this data come from?" derivation: a DECLARED catalog property appears iff something reads it from any registry slot (`referencingSlotsOf`) and no field or case operation writes it (`declarersOf`), standard properties and `case_id` excluded. Deliberately a FACT, never a finding — a no-writer read is a normal state (a viewer app over externally-managed cases is exactly this), so nothing gates, blocks, or asks for a fix. Memoized per doc reference like `effectiveCaseTypes`. Three consumers: the app-settings data-sources dialog (via `unwrittenPropertyCards` / `useUnwrittenPropertyCards` — the pre-rendered display projection), the blueprint summary's closing `<system_reminder>` block, and `getField`'s per-field reminder (`unwrittenPropertiesReadBy`). Read phrases render per audience (`ReadPhraseAudience`): the reminders speak the SA tool vocabulary (case-list column, search input, case-list filter), the dialog speaks the case workspace's authoring vocabulary (Cases available, search field, case information) with the builder's curly quotes — and the builder column phrases stay visibility-neutral because the index records only the slot, never which column or whether it is on-screen. Writer EXISTENCE is the whole check; never strengthen the copy past what is checked.
- `caseTypeRetirement.ts` — the orphaned case-type-record cascade every module-displacing surface consults: `planCaseTypeRetirementOnRemove` / `planCaseTypeRetirementOnRetype` decide, for a `removeModule` or a module case-type change, whether the displaced type's record retires in the same batch (orphaned + unreferenced), needs nothing (still owned), or blocks the batch with a person-readable reference list. Consumed by the SA/MCP `removeModule` + `updateModule` tools and this package's `useBlueprintMutations` — the cascade is an explicit granular `retireCaseType` mutation from the batch-building layer, NEVER a reducer side effect, so historical event-log replay reduces old `removeModule` events to the same docs it always did (and a concurrent edit to a DIFFERENT type merges rather than being clobbered).
- `caseOperationMutations.ts` — semantic add/update/remove/move planners for dormant form case operations. Remove blocks while any target, link, expression, or predicate has an `id-of` edge to the operation. Move blocks dependency inversion, target-type-transition inversion, possible-runtime-alias inversion, and multiplicity-scope inversion. `caseOperationOrder.ts` owns the shared dependency/conditional-guard analysis plus the representable physical sequence: the root container first, then exact repeat templates in post-order field traversal; authored-ID creates precede every non-create because HQ sorts a colliding case's create first. A repeated authored-key create and a later possibly-aliasing non-create also may not share a repeated execution ancestor: Core walks `C1,U1,C2,U2`, but HQ groups the duplicate concrete id and stable-sorts it to `C1,C2,U1,U2`. Root sibling repeats remain independent batches, and a provably distinct target remains legal. Conditional create/retype facts carry their predicate guards transitively to every later exact identity/type consumer. The same analysis keeps a runtime expression identity's immutable pre-submission lookup type separate from its rolling semantic type after retype, and conservatively rejects a later differently-typed target/link that may resolve to an earlier transitioned concrete id. It proves only different operation-created identities, a fresh generated create versus pre-submission data, and unequal literal ids distinct. The proof extends through ordinary actions emitted after advanced operations: a primary property write or child-case parent link requires the loaded session case to retain the module type; a write-free close action is type-agnostic. Transition history stays explicit because a conditional restoration cannot erase the earlier branch where it did not run. Repeated retype additionally requires an exact same-repeat generated create; duplicate authored keys or expression/session ids are not unique per iteration. The wire emitter and future runtime consume those projections; no planner may promise an order, branch, or target type the submitted XML cannot preserve.
- The reference index registers every operation's case types, repeat, form-field leaves, operation-id leaves, targets, links, and property declarations. Operation writes therefore participate in `declarersOf`; property renames rewrite both write keys and structural AST leaves, and case-type retirement reports operation references rather than orphaning their catalog entries.
- `connectConfig.ts` — connect-config doc helpers: `normalizeConnectConfig` (empty-sub-config stripper, consumed by the agent's mutation builders in `lib/agent/blueprintHelpers.ts`) and `dedupeRestoredConnectIds` (forces ids unique at the source when the form-settings Connect toggles restore or seed a block — the UI twin of the agent path's `enforceConnectIds`). Wire-emit defaults for `deliver_unit.entity_id` / `entity_name` live separately at `lib/commcare/connectDefaults.ts`.
- `expressionText.ts` — the doc-aware text ⇄ AST bridge every commit surface shares: `parseXPathForField` / `parseXPathForForm` turn authored XPath TEXT into the stored expression AST (`lib/domain/xpath`), resolving form-local references to identity leaves against the doc of the moment; `resolveCloseFieldRef` is the close-condition twin (authored field id → the target's uuid, pre-order first match — the wire emitter's `findField` rule). Parsing is TOTAL — an unresolved reference stays a raw leaf and the commit gate adjudicates the PRINTED text with the same validator findings as ever, so there is no separate parse-failure channel. Display is the inverse projection: `hooks/useXPathSlots.ts` prints reactively (a rename re-renders the shown text) and parses at commit.
- `expressionMigration.ts` — the one-time string → AST converter the scan/migrate scripts (`scripts/scan-expression-asts.ts` / `migrate-expression-asts.ts`) and the byte-identity proofs drive. Round-trip-gated per slot (`print(parse(s)) === s` or the slot is reported and left as text); event payloads convert against the doc reduced forward from the app's own stream. Steady-state code never calls it.
- `identifierVerdicts.ts` — the shared field-id verdicts every commit surface consults pre-dispatch (XML element-name legality, the reserved `__nova_` prefix, the case-property length cap, sibling-id uniqueness — peer-aware for renames). One verdict, every caller: the `FieldIdentitySection` rename guard, this package's `useBlueprintMutations.renameField` pre-check, and the SA/MCP `addFields` + `editField` tools; the validator's `DUPLICATE_FIELD_ID` / `INVALID_FIELD_ID` / `RESERVED_FIELD_ID_PREFIX` rules stay as backstops. Reducers never call it — they stay total so historical event-log replay can't block.
- `commitVerdicts.ts` — the identifierVerdicts pattern generalized to the whole validator: `prepareMutationCandidate` applies once and captures the candidate, reducer results, and scope; `evaluatePreparedMutationCandidate` runs the introduced-error gate without access to the mutations; `mutationCommitVerdict(prevDoc, mutations, lookupContext)` is the convenience composition. Previous and candidate validation always share the exact same external lookup-definition context. `describeIntroducedErrors` renders a rejection's findings as one person-to-person message. One rule everywhere: a batch may not introduce a shape/soundness/completeness finding, whatever the app's state. Three consumers: the SA/MCP tool layer (`lib/agent/tools/common.ts::guardedMutate`), this package's `useBlueprintMutations` (every dispatch gated — a rejected edit never reaches the store; call sites that present the returned findings contextually dispatch through the hook's `inline` flavor, all others announce via `notifyRejectedCommit`'s error toast), and the session store's `switchConnectMode` (the app-level Connect flip commits as one gated batch carrying each participating form's block). Undo/redo, hydration, the agent stream, and replay write through other paths and bypass the gate — they replay already-committed states. The client-only case-workspace status projection passes the explicit unavailable lookup context because it does not own a Project definition snapshot; structural lookup findings still mark the corresponding Search, Results, or Details surface broken, and its cache includes lookup-context identity so a future context-owning caller cannot reuse a stale projection. Do not add client fetching merely to strengthen this status projection.
- `dormantLookupCarriers.ts` — the Nova-neutral inventory behind S05a's closed commit/export gates. It fingerprints the complete authored Predicate / ValueExpression slot that contains a table carrier, so changing an operator or peer literal cannot hide behind stable table/column ids; a select source fingerprints `optionsSource` alone, deliberately excluding the inline receiver fallback. Stable sub-entity owners keep independent columns, search inputs, operations, and fields distinct. Operation write/link members use their authored property/identifier key as a stable logical subpath because they have no UUID, so sibling reorders do not rename an unchanged carrier. A historical unchanged fingerprint remains repairable through unrelated edits.
- `lookupReferences.ts` — the client-safe structural seam for dormant lookup carriers: exact occurrences (carrier uuid + immutable registry slot + canonical nested subpath + stable table/column ids + optional nonempty accepted-column-type set), the required available/unavailable validation context, and sorted/deduplicated target sets where every column implies its table. S05a's frozen production registry covers select sources and every Predicate / ValueExpression carrier owned by modules, forms, calculated columns, search inputs, and case operations. Operation write/link occurrences use their validator-enforced unique property/identifier as the canonical member anchor, never array position, so sibling reorders preserve structural-finding identity. Synthetic tests may still pass an explicit immutable registry. Do not add lookup buckets to `referenceIndex.ts` before a real consumer lands.
- `referenceIndex.ts` — the reference + declarations index behind `BlueprintDoc.refIndex` (shape + key vocabulary in `lib/domain/referenceIndex.ts`). Every write-path reference operation answers "who references X / who declares X" through its queries (`referencingCarrierUuids`, `declarersOf`) instead of walking the doc — a write-path reference scan is a defect. The index carries only structure something consumes — an unconsumed bucket maintained on every mutation is pure cost, so when a query loses its last consumer the bucket leaves both builders with it. Edges key on identity (target uuid / `(caseType, property)` / case-type name) and record (carrier uuid, registry slot id), never character spans: a consumer that needs structure walks the named slot's AST leaves or re-locates its prose hashtags. Module/form `displayCondition` predicates are registered as `module_display_condition` / `form_display_condition`; property renames structurally rewrite them and case-type retirement reports their references in the owning entity's voice. `applyMutation(s)` seeds the index on first contact and maintains it per mutation (mid-batch current — reducers read it), so every apply surface carries an identical index; hydration boundaries (`store.load`, `loadAppBlueprint`, the chat route's working doc) build it eagerly; `toPersistableDoc` strips it — it never serializes anywhere. Correctness is held by the incremental ≡ rebuild oracle: `buildReferenceIndex` doubles as the hydration builder and the fuzz oracle (`__tests__/referenceIndex.fuzz.test.ts` over raw mutation batches, plus the construction fuzz's parity assertion over real tool sequences), with a throttled dev-mode tripwire at batch end. Consumers: the rename/move rewriters in `mutations/fields.ts`, `caseTypeRetirement.ts`'s reference scan, `identifierVerdicts.ts`'s peer-aware conflict scan, and `unwrittenProperties.ts`'s read + writer lookups (`referencingSlotsOf` / `declarersOf`). The chip resolve gate deliberately stays catalog-backed — the catalog is a superset of writer declarations (a type is declared explicitly via the `declareCaseType` chokepoint, and `ensureCatalogProperty` then appends a writer's property to that EXISTING declared type — it does not mint an absent type, which is what lets two members add different properties to one type and merge), and the validator's accept set reads the same catalog, so re-keying chips to live writers would create offer-then-reject drift.
- `types.ts` — the `PersistableDoc` serialized shape.
