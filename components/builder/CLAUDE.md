# Builder Components

## BuilderLayout

Main layout with one `useChat` instance targeting `/api/chat`. Wrapped in `ErrorBoundary` around ChatSidebar (right), StructureSidebar (left), PreviewShell, and ContextualEditor.

- **`body`** sends: `apiKey`, `blueprint` (for edits), `projectId` (for Firestore persistence).
- **`sendAutomaticallyWhen`** only triggers for `askQuestions` (client-side tool).
- **`onData`** handles all state updates via `applyDataPart()`. Intercepts `data-project-saved` to update the URL via `replaceState` (no navigation/remount).
- **Agent status sync** — `useEffect` syncs `useChat` status → `builder.setAgentActive()`, enabling `builder.isThinking` for both generation and edit operations.
- **`messages: persistedChatMessages`** seeds useChat with module-level cached messages on mount. The AI SDK's `Chat` instance lives in a `useRef` inside `useChat`, so it resets on component remount. `persistedChatMessages` (module-level, like the `Builder` singleton) bridges the gap — updated on every render, restored on remount.
- **Redirect guard** — `if (shouldRedirect) return null` early-exits when no API key is set. All hooks must be declared above this guard to satisfy React's rules of hooks.
- **Project loading** — when `buildId !== 'new'` and user is authenticated, `builder.startLoading()` fires in a `useLayoutEffect` before first paint, then fetches `GET /api/projects/{id}`. On success, `builder.loadProject(id, blueprint)` atomically transitions `Loading → Ready`. If `status !== 'complete'` (error or stale generating), redirects to `/builds` with a toast — prevents broken empty-blueprint hydration. The `Loading` phase is the single source of truth (shows pulsing Logo) — no separate `projectLoaded` React state. BYOK users skip the fetch.
- **Auto-save** — `useAutoSave(builder, isAuthenticated)` subscribes to blueprint mutations and debounces saves (2s) to `PUT /api/projects/{id}`. Returns `SaveState` (`{ status: SaveStatus, savedAt: number | null }`) — consumed by `SaveIndicator` in the subheader. Status transitions: `idle` → `saving` (immediate on mutation) → `saved` (after successful PUT) or `error`. Tracks `builder.mutationCount` to avoid writes on selection-only changes. Reads all builder state live inside the callback to avoid stale closures.
- **Header coordination** — The global `AppHeader` (in root layout) owns the logo bar and `HeaderNav`. BuilderLayout coordinates via `lib/stores/headerVisibility.ts`: `setHeaderVisible(false)` in centered mode, `setHeaderVisible(true)` when generation starts. A `useLayoutEffect` ensures the store update fires before paint so React batches the header reveal with the hero Logo unmount. During centered mode, BuilderLayout renders a floating `HeaderNav` overlay (`absolute top-3 right-4 z-raised`) since the global header is collapsed. When the header reveals, the floating nav unmounts — only one `HeaderNav` instance at a time.

### Layout Structure

1. **Global header** (owned by root layout): `AppHeader` handles the logo bar + `HeaderNav`. On builder routes, uses CSS `grid-template-rows: 0fr/1fr` transition (450ms) and `motion.div layoutId="nova-logo"` on the Logo for shared layout animation with the builder's hero Logo.
2. **Builder subheader**: Nav buttons + `CollapsibleBreadcrumb` (left) + action buttons (right: SaveIndicator, Connect, Undo, Redo, Export). `h-12 bg-nova-deep`. Shows as soon as centered mode exits. Nav buttons appear when blueprint exists. Right-side action buttons appear when `showToolbar` (`Ready` + blueprint). `SaveIndicator` shows auto-save status with a ticking relative timestamp (cloud icon + "Saved Xm ago"); hidden until the first save occurs. Undo/Redo are icon-only (`w-8 h-8`). Export uses `ExportDropdown compact` (icon-only trigger with dropdown).
3. **Content area**: Flex row of sidebars and main content. StructureSidebar (left, `w-80`) and ChatSidebar (right, `w-[280px]`) are flex siblings of the main content — not floating overlays. Both animate width on open/close via `SIDEBAR_TRANSITION` (200ms, shared config). StructureSidebar uses `AnimatePresence` + `motion.div` for mount/unmount width animation. ChatSidebar stays always-mounted inside a `motion.div` wrapper that animates width (0 when collapsed) to preserve its singleton controller, scroll state, and observers. In centered mode the wrapper is transparent (`width: auto`, `duration: 0`, no overflow clip). `CursorModeSelector variant="vertical"` (vertical icon-only buttons) floats at the right edge of the main content area. ContextualEditor floats as a portal anchored to the selected question. When collapsed, floating reopen buttons appear at `absolute top-3 left-3/right-3` inside the main content scroll area.

### Chat-Centered Landing

When `builder.phase === Idle`, chat fills center with hero Logo above welcome heading and input — global header collapsed, floating `HeaderNav` overlay visible, uniform `bg-nova-void`. On generation start, hero Logo animates to the global header via `layoutId="nova-logo"` (global, no `LayoutGroup` — avoids triggering layout measurement passes that re-animate sibling panels), global header reveals via CSS grid transition, floating nav unmounts, builder subheader and panels fade in, chat becomes a 280px right sidebar flex child. Both sidebars start open (`useState(true)`). `AnimatePresence` fades in builder panels (150ms delay). No DOM re-parenting — messages and input state preserved.

### Builder Subheader

Single unified bar: nav buttons + `CollapsibleBreadcrumb` (left) + action buttons (right). `h-12 bg-nova-deep`. Builder-specific toolbar — separate from the global `AppHeader`. Shows as soon as centered mode exits. Right-side actions (`SaveIndicator`, `AppConnectSettings`, icon-only Undo/Redo, `ExportDropdown compact`) only render when `showToolbar`. `SaveIndicator` (`components/builder/SaveIndicator.tsx`) consumes `SaveState` from `useAutoSave` — shows "Saving…" with cloud-up icon immediately on mutation, then "Saved just now" / "Saved Xm ago" with cloud-check icon after success. Ticks every 15s to keep the relative timestamp current. Error state shows "Save failed" in rose with `role="alert"`. `AnimatePresence` handles mount/unmount only — status transitions update in place (no fade between states).

`CollapsibleBreadcrumb` (exported from `SubheaderToolbar.tsx`) renders navigable breadcrumbs, derived from the current screen's hierarchical position (via `nav.breadcrumbPath`). During generation (no blueprint yet), the app name appears as a static non-clickable breadcrumb via `builder.treeData?.app_name`. Once generation completes and `builder.blueprint` exists, normal nav-based breadcrumbs take over seamlessly. Follow-up forms show the selected case name as the final breadcrumb segment. Clicking a breadcrumb calls `nav.navigateTo()` + `syncSelection()`. Breadcrumbs use `text-lg whitespace-nowrap` — no truncation since the full-width bar has ample space. Collapse behind `…` dropdown only at depth 4+. **Breadcrumb labels update on inline title edits** — `usePreviewNav` splits breadcrumb computation: `breadcrumbPath` (screen structure) is memoized on `[current, blueprint]`, while `breadcrumb` (label strings) is derived unmemoized via `deriveBreadcrumbLabels()` so in-place blueprint mutations are reflected immediately on re-render. **Stable DOM structure** — uses a single `.map()` render path with `<button>` for every segment (styled differently for ancestor vs. current), preventing DOM teardown/rebuild when depth changes. **Memoized rendering** — click handlers are memoized on `nav.breadcrumbPath` in BuilderLayout, and `CollapsibleBreadcrumb` uses `memo` with deep part comparison (`breadcrumbPartsEqual`) to skip re-renders from unrelated state changes.

### Cursor Mode Bar

`CursorModeSelector variant="vertical"` (`CursorModeSelector.tsx`) — vertical stack of three icon-only buttons (pointer, text, inspect) with `title` tooltips. Anchored to the right edge of the centered form content via a ghost `max-w-3xl mx-auto` container (`-right-7` offset), so it maintains a fixed distance from form content regardless of sidebar state. Always visible when `showToolbar` (including pointer mode, so users can switch back). Each button has an animated mode-colored background indicator via `layoutId`. `CursorModeSelector` is a single component with `variant: 'horizontal' | 'vertical'` — horizontal shows icon + label in a segmented control, vertical shows icon-only with tooltips.

`usePreviewNav` is lifted to BuilderLayout and shared with `PreviewShell` (via `nav` prop) so navigation state stays in sync. PreviewShell renders with `hideHeader` since BuilderLayout owns the header tiers. `syncSelection(screen)` maps a `PreviewScreen` to the corresponding `builder.select()` call — used by both `handlePreviewBack` (history back, wired to Tier 2 nav buttons and passed to PreviewShell via `onBack` for post-submit navigation) and `handlePreviewUp` (hierarchy up via `getParentScreen`, wired to Tier 2 nav buttons).

### View Mode Sync

`viewMode` state (`'design' | 'preview'`). Two modes only — the app tree lives permanently in the StructureSidebar rather than as a separate "overview" mode.

- **Design ↔ Preview**: Nav is shared (no sync needed). Selection preserved but invisible in preview mode. Preview → Design preserves existing selection state — sidebar only opens if user had something selected before entering preview. Design → Preview auto-focuses the selected question's input. **Flipbook scroll sync**: `handleViewModeChange` captures the topmost visible question's `data-question-id` and its pixel offset from the scroll container top before calling `setState`. A `useLayoutEffect` on `viewMode` then restores the scroll position before paint — finding the same element (or the nearest visible question above it if the anchor is hidden by relevancy) and adjusting `scrollTop` to match the captured offset. The scroll container is identified via `[data-preview-scroll-container]` on PreviewShell's inner scroll div.
- **Escape in preview**: Switches to design without nav sync (stays on current screen).
- **Structure tree selection**: `handleTreeSelect` in BuilderLayout calls `builder.select()` and typed nav methods (`nav.navigateToForm()`, `nav.navigateToModule()`, `nav.navigateToHome()`). Typed methods are idempotent — they use `screensEqual()` internally to skip if already on the target screen, preventing duplicate history entries when clicking multiple questions in the same form.

When `Ready` + blueprint exists:
- `'design'` → `PreviewShell` (editable canvas) + `StructureSidebar` (left) + `ChatSidebar` (right) + `InlineSettingsPanel` (per-question, inspect mode only)
- `'preview'` → `PreviewShell` (read-only, no edit chrome, no sidebars)

### Panel State Management

Both sidebars are manually collapsible via close buttons in their headers. No auto-hide for any cursor mode or view state — visibility is purely user-controlled. InlineSettingsPanel shows when `cursorMode === 'inspect'` and a question is selected.
- `chatOpen` / `structureOpen` — both default to `true`, toggled by header close buttons and floating reopen buttons
- `showInlinePanel = isEditMode && cursorMode === 'inspect'` (rendered by FormRenderer per-question)

Both sidebars start open. After generation, the canvas auto-navigates to the first form. Both sidebars are independently open/closeable.

## StructureSidebar

Structure sidebar (`StructureSidebar.tsx`). Header with close chevron-left (left) + "Structure" label (right). Embeds `AppTree` with `hideHeader`. `onTreeSelect` prop delegates to BuilderLayout's `handleTreeSelect` which calls `builder.select()`, typed nav methods (`navigateToForm`/`navigateToModule`/`navigateToHome`), and scrolls the design canvas to the selected question if not already visible (250ms delay, visibility check, `block: 'start'`). Tree supports expand/collapse at module, form, and group/repeat levels.

**Indentation** — consistent 8px (`pl-2`) step per hierarchy level. Module headers at `pl-3` (12px), form headers at `pl-5` (20px), questions at `pl-7` (28px base + `depth * 8`px for nested groups/repeats). Indentation is applied as `paddingLeft` on each full-width row — no container-level margins — so alternating row backgrounds, hover highlights, and selection fills span the entire sidebar width regardless of nesting depth. Modules render full-width with `border-b` separators (no rounded card containers). Leaf questions omit the collapse chevron entirely (zero width), so their type icon aligns flush with the indent position; group/repeat questions show `[chevron] [icon] [name]`.

Question rows use `text-xs` font, alternating row backgrounds (pre-computed via `buildOddPaths`), and `data-tree-question` attributes for cross-panel scroll targeting. **Fuzzy search** input pinned above the scrollable area filters the tree by question label, question ID, module name, or form name. Uses fuse.js (`lib/filterTree.ts`) with `useDeferredValue` for responsive input. Matching branches preserve parent hierarchy; non-matching items are hidden. Parent nodes auto-expand via `forceExpand` set. Matched text highlighted with `bg-nova-violet/20`. Escape clears the search; state resets on sidebar close (AppTree unmounts). Container: `w-80 border-r border-nova-border-bright`. Flex sibling of main content, width-animated via `AnimatePresence` in BuilderLayout.

**Generation lock** — while `phase !== Ready`, the tree is non-interactive but still scrollable. A `pointer-events-none` `bg-black/25` overlay dims the tree content area (below the header). Individual clickable rows (module header, form header, question row) get `pointer-events-none` to block selection, with hover classes and `cursor-pointer` removed. Collapse chevrons use `invisible` to hide while preserving layout. The search input is disabled with reduced opacity. The close button in the header remains interactive. The overlay fades out (300ms) via `AnimatePresence` when generation completes. A `locked` prop threads from `AppTree` (derived from `phase !== Ready`) through `ModuleCard` → `FormCard` → `QuestionRow` → `CollapseChevron`. Edit mode is unaffected since edits only occur after `Ready`.

Keyboard shortcuts extracted to `useBuilderShortcuts.ts` hook, registered via `useKeyboardShortcuts`. Undo/redo shortcuts delegate to `handleUndo`/`handleRedo` callbacks (passed from BuilderLayout) rather than calling `builder.undo()`/`redo()` directly — this enables view restoration after undo/redo.

### Undo/Redo View Restoration

Undo/redo "teleports" the user back to where the edit was made. Each history snapshot captures the current `ViewMode`. On undo/redo, `builder.undo()`/`redo()` return the captured view mode. BuilderLayout's `restoreView()` switches viewMode if needed and syncs the preview nav via typed methods (`navigateToForm`/`navigateToModule`/`navigateToHome`) so the correct form is visible. `builder.setViewMode()` keeps the HistoryManager in sync on each render.

## InlineSettingsPanel

Inline property panel for inspect cursor mode (`cursorMode === 'inspect'`). Renders below the selected question inside the form DOM, pushing subsequent questions down and scrolling with the question. Drag-drop still works — the panel is inside `SortableQuestion` and moves during drag. **No self-scroll** — the panel does not scroll itself into view on mount; scroll-to-selection is owned by BuilderLayout's `handleTreeSelect` and `EditableQuestionWrapper` to avoid competing smooth-scroll animations.

All sub-editors share the `QuestionEditorProps` interface (`{ question, builder }`) from `contextual/shared.ts`. Each derives `selected`, `mb`, and `notifyBlueprintChanged` from `builder` internally. `useSaveQuestion(builder)` encapsulates the common field mutation + notify pattern.

Collapsible sections (all start expanded; Appearance hidden entirely for hidden questions):
- **Appearance** (`ContextualEditorUI`) — hint field and add-hint button. Label is intentionally absent — labels are edited directly on the canvas in text mode.
- **Logic** (`ContextualEditorLogic`) — required (Toggle + optional conditional XPath, hidden until added via "Add Property"), validation (with nested validation_msg), relevant, default_value, calculate. Standard XPath fields use a shared `XPathSection` component. `useAddableField` hook (from `shared.ts`) tracks pending fields added via "Add Property" — two instances: one for text fields, one for XPath fields.
- **Data** (`ContextualEditorData`) — question ID (with rename propagation), `CasePropertyPills` ("Saves to" header + pill buttons), options editor for select types

**Footer** (`ContextualEditorFooter`) — move up/down, type change, duplicate, delete. The type change button (center, icon + label + chevron) opens a `QuestionTypeGrid` upward (`placement: 'top-start'`) showing only valid conversion targets from `getConvertibleTypes()` (`lib/questionTypeConversions.ts`). Non-convertible types (hidden, label, geopoint, barcode) render a disabled button with a tooltip. Uses `flattenQuestionPaths` with `builder.mutationCount` dependency to keep move button enabled/disabled state fresh after mutations.

Reads/writes through `builder.mb` (persistent `MutableBlueprint`). Editing patterns:

- **useCommitField** (`hooks/useCommitField.ts`) — shared hook encapsulating the commit/cancel/checkmark model used by all DOM-based inline editors (`EditableText`, `InlineField` in FormSettingsPanel). Draft isolation while focused; Enter/blur commits; Escape cancels with `stopPropagation` (so the parent popover stays open); `committedRef` guard prevents double-commit when Enter triggers blur; `saved` is true for 1.5 s post-commit to drive the checkmark animation.
- **EditableText** — Always-rendered input/textarea styled as static text when unfocused, editing chrome on focus. Click places cursor at the native click position (no DOM swap). Single-line: Enter saves. Multiline: 1–4 row textarea sized to content (scrollable, doesn't auto-expand), Enter inserts newline, Cmd/Ctrl+Enter saves. Escape cancels (reverts to pre-edit value, `stopPropagation` prevents `useDismissRef` from closing the parent popover). Blur saves. Emerald checkmark fades on save (via `SavedCheck`). `autoFocus` focuses on mount. `selectAll` selects all text on focus (used for newly added questions via `builder.isNewQuestion(path)`, cleared on first save via `builder.clearNewQuestion()`). Optional `labelRight` renders right-aligned content in the label row. Label row order: label text, `SavedCheck`, shortcut hint (right via `ml-auto`), `labelRight`.
- **EditableTitle** (`components/builder/EditableTitle.tsx`) — Inline editable title for screen headers (app name, module name, form name). Renders an `<input>` styled as an `<h2>` — transparent border at rest, `border-nova-border` on hover, violet border on focus. Hidden `<span>` mirror sizes the input to exact text width. Calls `onSaved` callback on successful save so the parent controls checkmark placement. **`readOnly` prop** — renders the identical `<input>` element non-interactively (`pointer-events-none`, no handlers); used by all three screens in preview mode so the title occupies the exact same box as in design mode, preventing layout shift on toggle. Typography and box-model classes are extracted to module-level constants (`MEASURE_SPAN_CLASS`, `INPUT_BASE_CLASS`) so both paths are guaranteed to stay in sync. **SavedCheck** — shared animated emerald checkmark component (`size` prop, default 16; `className` prop, default `"shrink-0 -ml-1"` for title contexts). Used by EditableText (size 12), InlineField (size 10), and RefLabelInput (size 12) in addition to the three title screens.
- **AddPropertyButton** (`contextual/AddPropertyButton.tsx`) — Small pill button with "+" icon for adding optional properties/fields. Shared across Logic tab, UI tab, and OptionsEditor.
- **DropdownMenu** (`components/ui/DropdownMenu.tsx`) — Shared dropdown menu component. Renders icon + label rows with optional descriptions, optional violet dot + highlight for active selection (`activeKey`). First/last items inherit container border radius so hover/active backgrounds tile flush against rounded edges. `variant` prop controls the surface layer: `'glass'` (default, `POPOVER_GLASS` L1) for standalone popovers, `'elevated'` (`POPOVER_ELEVATED` L2) when stacked above an existing glass panel. Used by `ExportDropdown`, `FormTypeDropdown`, and `AfterSubmitSection`.
- **ExportDropdown** (`components/ui/ExportDropdown.tsx`) — Export format picker (Web/JSON, Mobile/CCZ). Compact mode: icon-only 32px trigger (`ci/share-ios-export`). Default mode: labeled button with chevron. Popover uses `DropdownMenu` with `AnimatePresence` entry/exit animation.
- **FormTypeButton** (`components/builder/detail/FormDetail.tsx`) — Form type icon in the FormScreen header. Mutation props (`mb`, `moduleIndex`, `formIndex`, `notifyBlueprintChanged`) are optional — when provided, renders as a clickable button that opens a `FormTypeDropdown` (via `useFloatingDropdown` + `DropdownPortal`) with violet dot + highlight via `DropdownMenu`; when omitted, renders as a static icon. FormScreen always renders `FormTypeButton`, passing mutation props only in edit mode. `FormDetail` is a read-only component that renders close case info when `form.close_case` is set — takes only `{ form }` as props. `FormSettingsPanel` retains Connect configuration and the After Submit destination picker. `AfterSubmitSection` uses `useFloatingDropdown` with `matchTriggerWidth` + `DropdownMenu variant="elevated"` (L2, portal-rendered to avoid parent overflow clipping). Child portal dropdowns inside a parent portal (like AfterSubmitSection inside FormSettingsPanel) must stop mousedown propagation on `DropdownPortal` to prevent the parent's document-level dismiss handler from closing both — and must omit `contentPopover` to avoid dismissing the parent via content popover coordination.
- **QuestionTypeGrid** (`components/builder/QuestionTypeGrid.tsx`) — shared 2-column icon+label grid for type selection. Used by `ContextualEditorFooter` (type change, with `types` prop for filtered conversion targets) and `QuestionTypePicker` (insertion point, full grid). Highlights active type with violet accent. Optional `types` prop shows only the specified types; omit for the default insertion grid. `variant` prop controls the surface layer: `'glass'` (default, `POPOVER_GLASS` L1) for standalone use, `'elevated'` (`POPOVER_ELEVATED` L2, opaque) when stacked above an existing glass popover.
- **CasePropertyDropdown** — select-style dropdown with "Saves to" header. Options: "None" (no persistence) + one entry per writable case type (module's own type + child types via `getModuleCaseTypes`). Uses `useFloatingDropdown` + `DropdownMenu` + `DropdownPortal` — same pattern as `AfterSubmitSection`. Cyan-highlighted label when a case type is selected. Disabled for media types, locked for `case_name`.
- **RefLabelInput** (`components/builder/RefLabelInput.tsx`) — TipTap-based label input used by text mode's inline label editing. Typing `#` triggers a two-phase autocomplete (namespace picker → filtered references from `ReferenceProvider`). Selected references insert as `commcareRef` atom nodes that render as inline `ReferenceChip` components. On load, `parseLabelSegments` splits bare `#type/path` hashtags into commcareRef nodes, building TipTap JSON content directly (no tiptap-markdown). On save, serializes commcareRef nodes back to bare `#type/path` hashtags. Debounced `onChange` (200ms) keeps the preview canvas in sync during editing. Same commit/cancel UX as `EditableText` (blur→save, Enter→save, Escape→cancel, `SavedCheck` emerald checkmark). Cannot use `useCommitField` because TipTap requires imperative editor commands (`editor.commands.blur()`, `editor.commands.setContent()`) rather than DOM ref calls. Keyboard shortcuts use two mechanisms: a TipTap `Extension` (`labelInputKeyboard`) for Enter/Mod-Enter (must run at ProseMirror priority to intercept before StarterKit's newline handler), and a DOM `keydown` listener for Escape (needs `stopPropagation`). Cancel reverts to a focus-time snapshot (`savedValueRef`) rather than the live `valueRef` (which gets updated by the debounced `onChange` mid-edit). TipTap extensions: `StarterKit` (paragraphs only), `CommcareRef` (custom atom node from `lib/tiptap/commcareRefNode.ts`), `labelInputKeyboard` (commit/cancel shortcuts), `Mention` (wired to `createRefSuggestion` from `lib/tiptap/refSuggestion.ts`).
- **ReferenceAutocomplete** (`components/builder/ReferenceAutocomplete.tsx`) — Floating autocomplete dropdown for hashtag reference suggestions. Imperative handle for keyboard navigation (`onKeyDown`). Two-phase display: namespace options (`#form/`, `#case/`, `#user/`) or filtered references with type-colored icons.
- **XPathField** — Dual-mode inline CodeMirror field. **Read-only** (no `onSave`): static syntax-highlighted display with reference chips, zero interactivity. **Editable** (`onSave` provided): click to activate a full inline editor with autocomplete, linting, bracket matching, and reference chips. Saves on blur, Escape, or Cmd+Enter; Escape dismisses the autocomplete dropdown first if open, then exits editing. Chips resolve automatically via the context `ReferenceProvider` (no prop wiring). `autoEdit` prop opens directly in edit mode (used for newly-added fields via "Add Property"). `onEditingChange` notifies parent when edit state changes. Used by `ContextualEditorLogic` for required/validation/relevant/default/calculate expressions and by `FormSettingsPanel` for Connect XPath fields.

### Content Popover Coordination

`hooks/useContentPopover.ts` — module-level singleton store that coordinates dismissal across floating UI popovers in the main content area. Content popovers (FormSettingsPanel, FormTypeDropdown, AppConnectPanel, QuestionTypePicker) live on the same visual layer and compete for focus. Each registers its dismiss callback via `useContentPopoverDismiss(onDismiss, enabled?)` on mount and unregisters on unmount. Consumers like `InsertionPoint` call `dismissContentPopovers()` to clear all active popovers before opening their own UI.

**Participants:** `FormSettingsButton`, `FormTypeButton`, `AppConnectSettings` (all pass `contentPopover: true` to `useFloatingDropdown`, which registers while open via `useContentPopoverDismiss(close, open)`), and `QuestionTypePicker` (calls `useContentPopoverDismiss` directly since it doesn't use the dropdown hook). The hook handles both content popover coordination and click-outside/Escape dismiss internally — no manual `useDismissRef` needed in portal-based dropdown consumers.

### Rename Propagation

- Editing question ID → `mb.renameQuestion()` → `rewriteXPathRefs` (Lezer-based) updates `/data/...` paths and `#form/...` hashtags across siblings.
- Renaming a case property (question with `case_property_on`) → `mb.renameCaseProperty()` → renames question ID across all forms in the module, `rewriteHashtagRefs` for `#case/...` refs, updates columns. Does not touch `case_types` (frozen after generation).

## GenerationProgress

Progress bar with stage labels during generation. Takes `stage` (`GenerationStage`) and `generationError` (`GenerationError`) props instead of reading phase directly. Always centered in the content area throughout the entire generation lifecycle — stays centered even after the scaffold tree appears in the structure sidebar. Mounted only while `isGenerating` or `generationError?.severity === 'failed'` — never shown for hydrated projects that jump straight to `Ready`. Exit animation (1s fade-out + slide-down) is handled by the parent `AnimatePresence` wrapper in BuilderLayout, not by the component itself. Counts derived from `builder.progressCompleted` / `builder.progressTotal`. Status message text was removed — stage-specific text now lives in the SignalGrid panel's etched label instead.

**Error state:** When `generationError` is set with `severity: 'failed'`, tracks the last active generating stage via `useRef` to show which step failed. The formerly-active stage gets an `'error'` status: static rose dot (no pulse), label in `text-nova-rose`, progress bar gradient shifts to `cyan→rose`. Error message from `generationError.message` displayed below the bar. No auto-dismiss on error — user must dismiss manually or retry.

## ReplayController

Fixed-position pill for log replay navigation. Left/right arrows, stage counter, close button. Drives Builder via `applyDataPart()`. Accepts `initialIndex` prop — defaults to 0 but BuilderLayout passes `doneIndex` so replay starts at the completed app state (edit stages after generation completes are still navigable forward).
