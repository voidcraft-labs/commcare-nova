# Builder Components

## BuilderLayout

Main layout with one `useChat` instance targeting `/api/chat`. Wrapped in `ErrorBoundary` around ChatSidebar (right), StructureSidebar (left), PreviewShell, and ContextualEditor.

- **`body`** sends: `apiKey`, `blueprint` (for edits), `projectId` (for Firestore persistence).
- **`sendAutomaticallyWhen`** only triggers for `askQuestions` (client-side tool).
- **`onData`** handles all state updates via `applyDataPart()`. Intercepts `data-project-saved` to update the URL via `replaceState` (no navigation/remount).
- **Agent status sync** ‚Äî `useEffect` syncs `useChat` status ‚Üí `builder.setAgentActive()`, enabling `builder.isThinking` for both generation and edit operations.
- **`messages: persistedChatMessages`** seeds useChat with module-level cached messages on mount. The AI SDK's `Chat` instance lives in a `useRef` inside `useChat`, so it resets on component remount. `persistedChatMessages` (module-level, like the `Builder` singleton) bridges the gap ‚Äî updated on every render, restored on remount.
- **Redirect guard** ‚Äî `if (shouldRedirect) return null` early-exits when no API key is set. All hooks must be declared above this guard to satisfy React's rules of hooks.
- **Project loading** ‚Äî when `buildId !== 'new'` and user is authenticated, fetches `GET /api/projects/{id}` on mount. If `status === 'complete'`, hydrates via `builder.reset()` ‚Üí `builder.setDone()`. If `status !== 'complete'` (error or stale generating), redirects to `/builds` with a toast ‚Äî prevents broken empty-blueprint hydration. Gates rendering with `projectLoaded` state (shows pulsing Logo). BYOK users skip the fetch.
- **Auto-save** ‚Äî `useAutoSave(builder, isAuthenticated)` subscribes to blueprint mutations and debounces saves (2s) to `PUT /api/projects/{id}`. Tracks `builder.mutationCount` to avoid writes on selection-only changes. Reads all builder state live inside the callback to avoid stale closures.
- **Header nav** ‚Äî uses `HeaderNav` from `components/ui/HeaderNav.tsx` (shared client component with active state and AccountMenu). Rendered once at a time ‚Äî either inside the collapsible header bar (non-centered) or as a floating overlay (centered mode). `isAdmin` passed from `useAuth()`.

### Three-Tier Header Layout

1. **Tier 1 ‚Äî Logo bar**: `commcare nova` + `HeaderNav` (Projects/Admin links with active state + `AccountMenu`). `bg-nova-void`. Collapses to zero height in hero/centered mode via CSS `grid-template-rows: 0fr/1fr` transition (no motion needed). Only one `HeaderNav` instance at a time ‚Äî header bar when visible, floating overlay when centered.
2. **Tier 2 ‚Äî Subheader**: Nav buttons + `CollapsibleBreadcrumb` (left) + icon-only action buttons (right: Connect, Undo, Redo, Export). `h-12 bg-nova-deep`. Shows as soon as centered mode exits. Nav buttons appear when blueprint exists. Right-side action buttons appear when `showToolbar` (`Done` + blueprint). Undo/Redo are icon-only (`w-8 h-8`). Export uses `ExportDropdown compact` (icon-only trigger with dropdown).
3. **Tier 3 ‚Äî Content area**: Flex row of sidebars and main content. StructureSidebar (left, `w-80`) and ChatSidebar (right, `w-[280px]`) are flex siblings of the main content ‚Äî not floating overlays. Both animate width on open/close via `SIDEBAR_TRANSITION` (200ms, shared config). StructureSidebar uses `AnimatePresence` + `motion.div` for mount/unmount width animation. ChatSidebar stays always-mounted inside a `motion.div` wrapper that animates width (0 when collapsed) to preserve its singleton controller, scroll state, and observers. In centered mode the wrapper is transparent (`width: auto`, `duration: 0`, no overflow clip). `CursorModeSelector variant="vertical"` (vertical icon-only buttons) floats at the right edge of the main content area. ContextualEditor floats as a portal anchored to the selected question. When collapsed, floating reopen buttons appear at `absolute top-3 left-3/right-3` inside the main content scroll area.

### Chat-Centered Landing

When `builder.phase === Idle && !builder.treeData`, chat fills center with hero Logo above welcome heading and input ‚Äî no header bars, uniform `bg-nova-void`. On generation start, Logo animates from center to header via `layoutId="nova-logo"` (global, no `LayoutGroup` ‚Äî avoids triggering layout measurement passes that re-animate sibling panels), header tiers slide in, chat becomes a 280px right sidebar flex child. Both sidebars start open (`useState(true)`). `AnimatePresence` fades in builder panels (150ms delay). No DOM re-parenting ‚Äî messages and input state preserved.

### Subheader (Tier 2)

Single unified bar: nav buttons + `CollapsibleBreadcrumb` (left) + icon-only action buttons (right). `h-12 bg-nova-deep`. Shows as soon as centered mode exits. Right-side actions (`AppConnectSettings`, icon-only Undo/Redo, `ExportDropdown compact`) only render when `showToolbar`.

`CollapsibleBreadcrumb` (exported from `SubheaderToolbar.tsx`) renders navigable breadcrumbs, derived from the current screen's hierarchical position (via `nav.breadcrumbPath`). During generation (no blueprint yet), the app name appears as a static non-clickable breadcrumb via `builder.treeData?.app_name`. Once generation completes and `builder.blueprint` exists, normal nav-based breadcrumbs take over seamlessly. Follow-up forms show the selected case name as the final breadcrumb segment. Clicking a breadcrumb calls `nav.navigateTo()` + `syncSelection()`. Breadcrumbs use `text-lg whitespace-nowrap` ‚Äî no truncation since the full-width bar has ample space. Collapse behind `‚Ä¶` dropdown only at depth 4+. **Breadcrumb labels update on inline title edits** ‚Äî `usePreviewNav` splits breadcrumb computation: `breadcrumbPath` (screen structure) is memoized on `[current, blueprint]`, while `breadcrumb` (label strings) is derived unmemoized via `deriveBreadcrumbLabels()` so in-place blueprint mutations are reflected immediately on re-render. **Stable DOM structure** ‚Äî uses a single `.map()` render path with `<button>` for every segment (styled differently for ancestor vs. current), preventing DOM teardown/rebuild when depth changes. **Memoized rendering** ‚Äî click handlers are memoized on `nav.breadcrumbPath` in BuilderLayout, and `CollapsibleBreadcrumb` uses `memo` with deep part comparison (`breadcrumbPartsEqual`) to skip re-renders from unrelated state changes.

### Cursor Mode Bar

`CursorModeSelector variant="vertical"` (`CursorModeSelector.tsx`) ‚Äî vertical stack of three icon-only buttons (pointer, text, inspect) with `title` tooltips. Anchored to the right edge of the centered form content via a ghost `max-w-3xl mx-auto` container (`-right-7` offset), so it maintains a fixed distance from form content regardless of sidebar state. Always visible when `showToolbar` (including pointer mode, so users can switch back). Each button has an animated mode-colored background indicator via `layoutId`. `CursorModeSelector` is a single component with `variant: 'horizontal' | 'vertical'` ‚Äî horizontal shows icon + label in a segmented control, vertical shows icon-only with tooltips.

`usePreviewNav` is lifted to BuilderLayout and shared with `PreviewShell` (via `nav` prop) so navigation state stays in sync. PreviewShell renders with `hideHeader` since BuilderLayout owns the header tiers. `syncSelection(screen)` maps a `PreviewScreen` to the corresponding `builder.select()` call ‚Äî used by both `handlePreviewBack` (history back, wired to Tier 2 nav buttons and passed to PreviewShell via `onBack` for post-submit navigation) and `handlePreviewUp` (hierarchy up via `getParentScreen`, wired to Tier 2 nav buttons).

### View Mode Sync

`viewMode` state (`'design' | 'preview'`). Two modes only ‚Äî the app tree lives permanently in the StructureSidebar rather than as a separate "overview" mode.

- **Design ‚Üî Preview**: Nav is shared (no sync needed). Selection preserved but invisible in preview mode. Preview ‚Üí Design preserves existing selection state ‚Äî sidebar only opens if user had something selected before entering preview. Design ‚Üí Preview auto-focuses the selected question's input. **Flipbook scroll sync**: `handleViewModeChange` captures the topmost visible question's `data-question-id` and its pixel offset from the scroll container top before calling `setState`. A `useLayoutEffect` on `viewMode` then restores the scroll position before paint ‚Äî finding the same element (or the nearest visible question above it if the anchor is hidden by relevancy) and adjusting `scrollTop` to match the captured offset. The scroll container is identified via `[data-preview-scroll-container]` on PreviewShell's inner scroll div.
- **Escape in preview**: Switches to design without nav sync (stays on current screen).
- **Structure tree selection**: `handleTreeSelect` in BuilderLayout calls `builder.select()` and typed nav methods (`nav.navigateToForm()`, `nav.navigateToModule()`, `nav.navigateToHome()`). Typed methods are idempotent ‚Äî they use `screensEqual()` internally to skip if already on the target screen, preventing duplicate history entries when clicking multiple questions in the same form.

When `Done` + blueprint exists:
- `'design'` ‚Üí `PreviewShell` (editable canvas) + `StructureSidebar` (left) + `ChatSidebar` (right) + `InlineSettingsPanel` (per-question, inspect mode only)
- `'preview'` ‚Üí `PreviewShell` (read-only, no edit chrome, no sidebars)

### Panel State Management

Both sidebars are manually collapsible via close buttons in their headers. No auto-hide for any cursor mode or view state ‚Äî visibility is purely user-controlled. InlineSettingsPanel shows when `cursorMode === 'inspect'` and a question is selected.
- `chatOpen` / `structureOpen` ‚Äî both default to `true`, toggled by header close buttons and floating reopen buttons
- `showInlinePanel = isEditMode && cursorMode === 'inspect'` (rendered by FormRenderer per-question)

Both sidebars start open. After generation, the canvas auto-navigates to the first form. Both sidebars are independently open/closeable.

## StructureSidebar

Structure sidebar (`StructureSidebar.tsx`). Header with close chevron-left (left) + "Structure" label (right). Embeds `AppTree` with `hideHeader`. `onTreeSelect` prop delegates to BuilderLayout's `handleTreeSelect` which calls `builder.select()`, typed nav methods (`navigateToForm`/`navigateToModule`/`navigateToHome`), and scrolls the design canvas to the selected question if not already visible (250ms delay, visibility check, `block: 'start'`). Tree supports expand/collapse at module, form, and group/repeat levels.

**Indentation** ‚Äî consistent 8px (`pl-2`) step per hierarchy level. Module headers at `pl-3` (12px), form headers at `pl-5` (20px), questions at `pl-7` (28px base + `depth * 8`px for nested groups/repeats). Indentation is applied as `paddingLeft` on each full-width row ‚Äî no container-level margins ‚Äî so alternating row backgrounds, hover highlights, and selection fills span the entire sidebar width regardless of nesting depth. Modules render full-width with `border-b` separators (no rounded card containers). Leaf questions omit the collapse chevron entirely (zero width), so their type icon aligns flush with the indent position; group/repeat questions show `[chevron] [icon] [name]`.

Question rows use `text-xs` font, alternating row backgrounds (pre-computed via `buildOddPaths`), and `data-tree-question` attributes for cross-panel scroll targeting. **Fuzzy search** input pinned above the scrollable area filters the tree by question label, question ID, module name, or form name. Uses fuse.js (`lib/filterTree.ts`) with `useDeferredValue` for responsive input. Matching branches preserve parent hierarchy; non-matching items are hidden. Parent nodes auto-expand via `forceExpand` set. Matched text highlighted with `bg-nova-violet/20`. Escape clears the search; state resets on sidebar close (AppTree unmounts). Container: `w-80 border-r border-nova-border-bright`. Flex sibling of main content, width-animated via `AnimatePresence` in BuilderLayout.

**Generation lock** ‚Äî while `phase !== Done`, the tree is non-interactive but still scrollable. A `pointer-events-none` `bg-black/25` overlay dims the tree content area (below the header). Individual clickable rows (module header, form header, question row) get `pointer-events-none` to block selection, with hover classes and `cursor-pointer` removed. Collapse chevrons use `invisible` to hide while preserving layout. The search input is disabled with reduced opacity. The close button in the header remains interactive. The overlay fades out (300ms) via `AnimatePresence` when generation completes. A `locked` prop threads from `AppTree` (derived from `phase !== Done`) through `ModuleCard` ‚Üí `FormCard` ‚Üí `QuestionRow` ‚Üí `CollapseChevron`. Edit mode is unaffected since edits only occur after `Done`.

Keyboard shortcuts extracted to `useBuilderShortcuts.ts` hook, registered via `useKeyboardShortcuts`. Undo/redo shortcuts delegate to `handleUndo`/`handleRedo` callbacks (passed from BuilderLayout) rather than calling `builder.undo()`/`redo()` directly ‚Äî this enables view restoration after undo/redo.

### Undo/Redo View Restoration

Undo/redo "teleports" the user back to where the edit was made. Each history snapshot captures the current `ViewMode`. On undo/redo, `builder.undo()`/`redo()` return the captured view mode. BuilderLayout's `restoreView()` switches viewMode if needed and syncs the preview nav via typed methods (`navigateToForm`/`navigateToModule`/`navigateToHome`) so the correct form is visible. `builder.setViewMode()` keeps the HistoryManager in sync on each render.

## InlineSettingsPanel

Inline property panel for inspect cursor mode (`cursorMode === 'inspect'`). Renders below the selected question inside the form DOM, pushing subsequent questions down and scrolling with the question. Drag-drop still works ‚Äî the panel is inside `SortableQuestion` and moves during drag. **No self-scroll** ‚Äî the panel does not scroll itself into view on mount; scroll-to-selection is owned by BuilderLayout's `handleTreeSelect` and `EditableQuestionWrapper` to avoid competing smooth-scroll animations.

Three collapsible sections (all start expanded; hidden questions start with Appearance collapsed):
- **Appearance** (`ContextualEditorUI`) ‚Äî type (via `QuestionTypeGrid` popover at `z-popover-top`), hint. Label is intentionally absent ‚Äî labels are edited directly on the canvas in text mode.
- **Logic** (`ContextualEditorLogic`) ‚Äî required (Toggle + optional conditional XPath, hidden until added via "Add Property"), validation, relevant, default_value, calculate. All XPath fields rendered via `XPathField` with automatic chip support via context provider.
- **Data** (`ContextualEditorData`) ‚Äî question ID (with rename propagation), `CasePropertyPills` ("Saves to" header + pill buttons), options editor for select types

**Footer** (`ContextualEditorFooter`) ‚Äî move up/down, duplicate, delete. Uses `flattenQuestionPaths` with `builder.mutationCount` dependency to keep move button enabled/disabled state fresh after mutations.

Reads/writes through `builder.mb` (persistent `MutableBlueprint`). Editing patterns:

- **useCommitField** (`hooks/useCommitField.ts`) ‚Äî shared hook encapsulating the commit/cancel/checkmark model used by all DOM-based inline editors (`EditableText`, `InlineField` in FormSettingsPanel). Draft isolation while focused; Enter/blur commits; Escape cancels with `stopPropagation` (so the parent popover stays open); `committedRef` guard prevents double-commit when Enter triggers blur; `saved` is true for 1.5 s post-commit to drive the checkmark animation.
- **EditableText** ‚Äî Always-rendered input/textarea styled as static text when unfocused, editing chrome on focus. Click places cursor at the native click position (no DOM swap). Single-line: Enter saves. Multiline: 1‚Äì4 row textarea sized to content (scrollable, doesn't auto-expand), Enter inserts newline, Cmd/Ctrl+Enter saves. Escape cancels (reverts to pre-edit value, `stopPropagation` prevents `useDismissRef` from closing the parent popover). Blur saves. Emerald checkmark fades on save (via `SavedCheck`). `autoFocus` focuses on mount. `selectAll` selects all text on focus (used for newly added questions via `builder.isNewQuestion(path)`, cleared on first save via `builder.clearNewQuestion()`). Optional `labelRight` renders right-aligned content in the label row. Label row order: label text, `SavedCheck`, shortcut hint (right via `ml-auto`), `labelRight`.
- **EditableTitle** (`components/builder/EditableTitle.tsx`) ‚Äî Inline editable title for screen headers (app name, module name, form name). Renders an `<input>` styled as an `<h2>` ‚Äî transparent border at rest, `border-nova-border` on hover, violet border on focus. Hidden `<span>` mirror sizes the input to exact text width. Calls `onSaved` callback on successful save so the parent controls checkmark placement. **`readOnly` prop** ‚Äî renders the identical `<input>` element non-interactively (`pointer-events-none`, no handlers); used by all three screens in preview mode so the title occupies the exact same box as in design mode, preventing layout shift on toggle. Typography and box-model classes are extracted to module-level constants (`MEASURE_SPAN_CLASS`, `INPUT_BASE_CLASS`) so both paths are guaranteed to stay in sync. **SavedCheck** ‚Äî shared animated emerald checkmark component (`size` prop, default 16; `className` prop, default `"shrink-0 -ml-1"` for title contexts). Used by EditableText (size 12), InlineField (size 10), and RefLabelInput (size 12) in addition to the three title screens.
- **AddPropertyButton** (`contextual/AddPropertyButton.tsx`) ‚Äî Small pill button with "+" icon for adding optional properties/fields. Shared across Logic tab, UI tab, and OptionsEditor.
- **DropdownMenu** (`components/ui/DropdownMenu.tsx`) ‚Äî Shared dropdown menu component. Renders icon + label rows with optional descriptions, optional violet dot + highlight for active selection (`activeKey`). First/last items inherit container border radius so hover/active backgrounds tile flush against rounded edges. `variant` prop controls the surface layer: `'glass'` (default, `POPOVER_GLASS` L1) for standalone popovers, `'elevated'` (`POPOVER_ELEVATED` L2) when stacked above an existing glass panel. Used by `ExportDropdown`, `FormTypeDropdown`, and `AfterSubmitSection`.
- **ExportDropdown** (`components/ui/ExportDropdown.tsx`) ‚Äî Export format picker (Web/JSON, Mobile/CCZ). Compact mode: icon-only 32px trigger (`ci/share-ios-export`). Default mode: labeled button with chevron. Popover uses `DropdownMenu` with `AnimatePresence` entry/exit animation.
- **FormTypeButton** (`components/builder/detail/FormDetail.tsx`) ‚Äî Form type icon in the FormScreen header. Mutation props (`mb`, `moduleIndex`, `formIndex`, `notifyBlueprintChanged`) are optional ‚Äî when provided, renders as a clickable button that opens a `FormTypeDropdown` (via `useFloatingDropdown` + `DropdownPortal`) with violet dot + highlight via `DropdownMenu`; when omitted, renders as a static icon. FormScreen always renders `FormTypeButton`, passing mutation props only in edit mode. `FormDetail` is a read-only component that renders close case info when `form.close_case` is set ‚Äî takes only `{ form }` as props. `FormSettingsPanel` retains Connect configuration and the After Submit destination picker. `AfterSubmitSection` uses `useFloatingDropdown` with `matchTriggerWidth` + `DropdownMenu variant="elevated"` (L2, portal-rendered to avoid parent overflow clipping). Child portal dropdowns inside a parent portal (like AfterSubmitSection inside FormSettingsPanel) must stop mousedown propagation on `DropdownPortal` to prevent the parent's document-level dismiss handler from closing both ‚Äî and must omit `contentPopover` to avoid dismissing the parent via content popover coordination.
- **QuestionTypeGrid** (`components/builder/QuestionTypeGrid.tsx`) ‚Äî shared 2-column icon+label grid for type selection. Used by both `ContextualEditorUI` (type change) and `QuestionTypePicker` (insertion point). Highlights active type with violet accent. `variant` prop controls the surface layer: `'glass'` (default, `POPOVER_GLASS` L1) for standalone use, `'elevated'` (`POPOVER_ELEVATED` L2, opaque) when stacked above an existing glass popover.
- **CasePropertyPills** ‚Äî pill buttons with "Saves to" header. One pill per case type the module can write to (its own type + child types via `getModuleCaseTypes`). Click to toggle on/off, radio behavior when multiple. Disabled for media types, locked on for `case_name`.
- **RefLabelInput** (`components/builder/RefLabelInput.tsx`) ‚Äî TipTap-based label input used by text mode's inline label editing. Typing `#` triggers a two-phase autocomplete (namespace picker ‚Üí filtered references from `ReferenceProvider`). Selected references insert as `commcareRef` atom nodes that render as inline `ReferenceChip` components. On load, `parseLabelSegments` splits bare `#type/path` hashtags into commcareRef nodes, building TipTap JSON content directly (no tiptap-markdown). On save, serializes commcareRef nodes back to bare `#type/path` hashtags. Debounced `onChange` (200ms) keeps the preview canvas in sync during editing. Same commit/cancel UX as `EditableText` (blur‚Üísave, Enter‚Üísave, Escape‚Üícancel, `SavedCheck` emerald checkmark). Cannot use `useCommitField` because TipTap requires imperative editor commands (`editor.commands.blur()`, `editor.commands.setContent()`) rather than DOM ref calls. Keyboard shortcuts use two mechanisms: a TipTap `Extension` (`labelInputKeyboard`) for Enter/Mod-Enter (must run at ProseMirror priority to intercept before StarterKit's newline handler), and a DOM `keydown` listener for Escape (needs `stopPropagation`). Cancel reverts to a focus-time snapshot (`savedValueRef`) rather than the live `valueRef` (which gets updated by the debounced `onChange` mid-edit). TipTap extensions: `StarterKit` (paragraphs only), `CommcareRef` (custom atom node from `lib/tiptap/commcareRefNode.ts`), `labelInputKeyboard` (commit/cancel shortcuts), `Mention` (wired to `createRefSuggestion` from `lib/tiptap/refSuggestion.ts`).
- **ReferenceAutocomplete** (`components/builder/ReferenceAutocomplete.tsx`) ‚Äî Floating autocomplete dropdown for hashtag reference suggestions. Imperative handle for keyboard navigation (`onKeyDown`). Two-phase display: namespace options (`#form/`, `#case/`, `#user/`) or filtered references with type-colored icons.
- **XPathField** ‚Äî Dual-mode inline CodeMirror field. **Read-only** (no `onSave`): static syntax-highlighted display with reference chips, zero interactivity. **Editable** (`onSave` provided): click to activate a full inline editor with autocomplete, linting, bracket matching, and reference chips. Saves on blur, Escape, or Cmd+Enter; Escape dismisses the autocomplete dropdown first if open, then exits editing. Chips resolve automatically via the context `ReferenceProvider` (no prop wiring). `autoEdit` prop opens directly in edit mode (used for newly-added fields via "Add Property"). `onEditingChange` notifies parent when edit state changes. Used by `ContextualEditorLogic` for required/validation/relevant/default/calculate expressions and by `FormSettingsPanel` for Connect XPath fields.

### Content Popover Coordination

`hooks/useContentPopover.ts` ‚Äî module-level singleton store that coordinates dismissal across floating UI popovers in the main content area. Content popovers (FormSettingsPanel, FormTypeDropdown, AppConnectPanel, QuestionTypePicker) live on the same visual layer and compete for focus. Each registers its dismiss callback via `useContentPopoverDismiss(onDismiss, enabled?)` on mount and unregisters on unmount. Consumers like `InsertionPoint` call `dismissContentPopovers()` to clear all active popovers before opening their own UI.

**Participants:** `FormSettingsButton`, `FormTypeButton`, `AppConnectSettings` (all pass `contentPopover: true` to `useFloatingDropdown`, which registers while open via `useContentPopoverDismiss(close, open)`), and `QuestionTypePicker` (calls `useContentPopoverDismiss` directly since it doesn't use the dropdown hook). The hook handles both content popover coordination and click-outside/Escape dismiss internally ‚Äî no manual `useDismissRef` needed in portal-based dropdown consumers.

### Rename Propagation

- Editing question ID ‚Üí `mb.renameQuestion()` ‚Üí `rewriteXPathRefs` (Lezer-based) updates `/data/...` paths and `#form/...` hashtags across siblings.
- Renaming a case property (question with `case_property_on`) ‚Üí `mb.renameCaseProperty()` ‚Üí renames question ID across all forms in the module, `rewriteHashtagRefs` for `#case/...` refs, updates columns. Does not touch `case_types` (frozen after generation).

## GenerationProgress

Progress bar with phase labels during generation. Always centered in the content area throughout the entire generation lifecycle ‚Äî stays centered even after the scaffold tree appears in the structure sidebar. Mounted only while `isGenerating` or `phase === Error` ‚Äî never shown for hydrated projects that jump straight to `Done`. Exit animation (1s fade-out + slide-down) is handled by the parent `AnimatePresence` wrapper in BuilderLayout, not by the component itself. Counts derived from `builder.progressCompleted` / `builder.progressTotal`. Status message text was removed ‚Äî phase-specific text now lives in the SignalGrid panel's etched label instead.

**Error state:** When `phase === BuilderPhase.Error`, tracks the last active generating phase via `useRef` to show which step failed. The formerly-active stage gets an `'error'` status: static rose dot (no pulse), label in `text-nova-rose`, progress bar gradient shifts to `cyan‚Üírose`. Error message from `statusMessage` prop displayed below the bar. No auto-dismiss on error ‚Äî user must dismiss manually or retry.

## ReplayController

Fixed-position pill for log replay navigation. Left/right arrows, stage counter, close button. Drives Builder via `applyDataPart()`. Accepts `initialIndex` prop ‚Äî defaults to 0 but BuilderLayout passes `doneIndex` so replay starts at the completed app state (edit stages after Done are still navigable forward).
